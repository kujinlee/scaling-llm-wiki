# AI 에이전트를 위한 Playwright E2E 테스트 하네스 구축하기 (Deep Dive)

**Channel:** NAVER D2 | **Duration:** 32:11 | **URL:** https://www.youtube.com/watch?v=wo0Rsh9hlTo

---

## 서론: AI가 쓴 코드, 누가 검증하나?
▶ [0:00–1:09](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=0s)
발표는 "AI가 쓴 코드 누가 검증하나? AI 에이전트를 위한 플레이라이트 E2E 테스트 하네스 구축하기"라는 제목으로 시작합니다. 발표자는 네이버 파이낸셜 금융 FE팀의 정회원입니다.

2026년은 AI 덕분에 개발 작업이 매우 편리해지는, 이른바 '딸깍의 시대'가 될 것이라고 예측합니다. 리팩토링, 버그 수정, 마이그레이션과 같은 작업들을 AI 에이전트에게 맡기면 되는 시대가 오고 있습니다. 하지만 현재 AI 에이전트가 작성한 코드는 완벽하지 않으며, 때로는 "덜컹거리는" 문제가 발생합니다.

한 가지 사례로, AI 에이전트가 Next.js 버전 업그레이드 작업을 성공적으로 완료했다고 보고하고 PR(Pull Request)까지 작성했지만, 실제 실행해보니 특정 페이지에서 뒤로 가기 버튼의 기존 동작이 깨져 있는 문제가 발견되었습니다. 이는 프레임워크 내부 구현 변경이 코드베이스에 영향을 미쳤기 때문인데, 에이전트는 이를 확인하지 않고 작업을 완료했다고 보고한 것입니다.

이러한 실수는 모델이 똑똑하지 않아서가 아니라, 검증할 방법이 주어지지 않았기 때문에 발생합니다. 따라서 모델이 일할 환경을 구축하는 **하네스 엔지니어링(Harness Engineering)**이 중요한 화두로 떠오르고 있습니다.

### 하네스 엔지니어링이란?
▶ [1:09–2:09](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=69s)
하네스 엔지니어링이라는 용어를 처음 정의한 미첼라 시모토(Michela Shimoto)는 이를 "에이전트가 실수할 때마다 그 실수가 다시는 반복되지 않도록 하는 것"이라고 설명했습니다. 구체적으로는 모델의 가중치(weights) 외부의 모든 것을 포함합니다.

- **지침 파일**: `agent.md`와 같은 지침 문서
- **도구**: `MCP 스킬스`와 같은 외부 도구
- **지식 기반**: 도메인 지식을 인코딩한 `LLM 위키`
- **검증 도구**: 린트, 자동화 테스트 등

이 발표에서는 이러한 하네스 중에서도 특히 **자동화 테스트**, 그중에서도 **E2E(End-to-End) 테스트**에 대해 깊이 있게 다룹니다.

## E2E 테스트의 필요성: 검증 병목 현상의 대두
▶ [2:09–2:29](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=129s)
다양한 하네스 중에서 E2E 테스트가 왜 중요한지에 대한 논의입니다. E2E 테스트는 작성하기 어렵고, 느리며, 잘 깨진다는 단점에도 불구하고 AI 에이전트 시대에 필수적인 요소가 되었습니다.

### 코딩 속도 증가와 검증의 병목
▶ [2:29–3:41](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=149s)
AI 에이전트 도입 이후 개발 속도는 확실히 빨라졌습니다. 발표자가 속한 팀의 코드베이스에서 한 개발자가 머지한 PR 수를 비교한 결과, AI 에이전트 도입 전 9개월간 227건이었던 것이 도입 후 9개월간 340건으로 약 **1.5배** 증가했습니다.

하지만 코딩 속도가 1.5배 빨라졌다고 해서 검증의 질도 1.5배 향상된 것은 아닙니다. 사람이 모든 PR을 로컬에서 직접 실행하며 일일이 검토하는 것은 현실적으로 불가능합니다. 이로 인해 **코딩은 빨라졌지만 검증이 병목**이 되는 현상이 발생했습니다.

클로드(Claude) 코드의 아버지로 알려진 보리스 체르니(Boris Cherny)는 "클로드에게 자기 작업을 검증할 수단을 줘라. 이건들 결과를 두세 배 끌어내는 방법이었다"라고 말하며, 모델에게 검증 수단을 제공하는 것의 중요성을 강조했습니다. 모델도 사람이 그렇듯 스스로 짠 코드에 대해서는 관대하기 때문에, 객관적인 검증 수단이 반드시 필요합니다.

### 단위 테스트의 한계와 컴포넌트 경계 결함
▶ [3:41–5:07](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=221s)
팀에서는 초기에 에이전트가 직접 브라우저를 조작하는 방식(Vercel의 Agent Browser, Playwright MCP 등)을 시도했습니다. 이 방식은 수동 테스트의 부담을 줄여주었지만, 매번 시나리오를 실행해야 해서 시간이 오래 걸리고, 토큰 소모가 많으며, 결과가 비결정적이라는 문제가 있었습니다.

따라서 결정론적인 테스트 코드가 필요하다는 결론에 도달했습니다. 기존 코드베이스에는 Vitest와 React Testing Library를 사용한 단위 테스트가 있었지만, AI가 코딩하는 시대에는 이것만으로 부족했습니다.

예를 들어, 미들웨어, `getServerSideProps`, 컴포넌트 각각의 단위 테스트는 모두 통과하더라도 실제 브라우저에서 실행하면 데이터 형식이 어긋나거나, 상태가 제대로 전달되지 않거나, 특정 환경에서 API 호출이 실패하는 등의 문제가 발생할 수 있습니다. 이는 각 단위는 정상이지만, 이들이 통합될 때 발생하는 **컴포넌트 경계 결함(Component Boundary Faults)** 때문이며, AI가 만든 오류의 대부분이 여기서 발생했습니다.

### 테스팅 트로피와 E2E 테스트의 중요성
▶ [5:07–6:27](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=307s)
켄트 C. 다즈(Kent C. Dodds)가 2018년에 제시한 '테스팅 트로피(Testing Trophy)' 모델은 테스트의 종류와 신뢰도의 관계를 설명합니다.

```ascii
+-----------------------------+
| End-to-End                  |  <-- 높은 비용, 가장 높은 확신
+-----------------------------+
| Integration                 |
+-----------------------------+
| Unit                        |
+-----------------------------+
| Static                      |  <-- 낮은 비용, 가장 낮은 확신
+-----------------------------+
```

- **Static (정적 분석)**: TypeScript, ESLint 등.
- **Unit (단위 테스트)**: 컴포넌트나 모듈을 고립시켜 검증.
- **Integration (통합 테스트)**: 여러 단위가 함께 동작하는 것을 검증.
- **End-to-End (E2E 테스트)**: 모킹(mocking) 없이 실제 브라우저에서 사용자 플로우 전체를 검증.

과거에는 E2E 테스트가 비용이 크고 깨지기 쉽다는 이유로 QA팀에 의존하는 경우가 많았지만, AI 에이전트 시대에는 테스트 작성 부담이 줄고 시스템 전체 검증의 중요도가 높아졌습니다. 마틴 파울러(Martin Fowler) 또한 E2E 테스트가 "소프트웨어가 작동하는지 가장 큰 확신을 준다"고 말했습니다. 이러한 배경에서 팀은 **Playwright로 E2E 테스트를 작성하기로 결정**했습니다.

## Playwright 도입: E2E 테스트 프레임워크 선택과 장점
▶ [6:27–6:37](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=387s)
팀에서 Playwright를 구축하며 느낀 장점은 다음과 같습니다.

### 1. 실제 브라우저와 사용자 수준의 검증
▶ [6:37–7:17](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=397s)
단위 테스트는 노드(Node.js)나 JSOM 환경에서 함수 단위의 로직을 검증하는 반면, Playwright는 **실제 사용자가 사용하는 브라우저 환경에서 사용자 경험(UX)을 검증**합니다. "버튼이 보인다", "클릭한다", "다음 화면으로 넘어간다"와 같이 실제 사용자 행동을 코드로 표현하며, 함수가 아닌 사용자 경험 자체를 테스트합니다.

### 2. 살아있는 명세(Living Specification)로서의 테스트 코드
▶ [7:17–8:05](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=437s)
Playwright 테스트 코드는 자연어처럼 읽히기 때문에 그 자체로 **사용자 명세(스펙 문서)**가 됩니다.

예시 코드:
```javascript
// 론 페이지에서 한도 조회 버튼을 클릭하면
await page.goto('/loan');
await page.getByRole('button', { name: '한도 조회' }).click();

// 한도 조회 결과 텍스트가 보인다.
await expect(page.getByText('한도 조회 결과')).toBeVisible();
```

이 코드는 실행하면 검증 도구(센서)가 되고, 읽으면 "사용자가 론 페이지에서 한도 조회 버튼을 누르면 한도 조회 결과가 보인다"는 명세(가이드)가 됩니다. 코드와 명세가 하나가 되어, 코드가 변경되면 명세도 함께 변경되므로 항상 최신 상태를 유지할 수 있습니다.

### 3. 에이전트를 위한 풍부한 디버깅 데이터 제공
▶ [8:05–10:17](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=485s)
Playwright는 테스트 실행 후 상세한 **HTML 리포트**를 제공합니다. 이 리포트에는 다음과 같은 정보가 포함됩니다.
- 테스트 목록 및 결과
- 스크린샷 및 실행 비디오
- 시간대별 화면 변화(Before/After 스크린샷)
- DOM 스냅샷
- 실행된 액션 목록
- 콘솔 로그 및 네트워크 요청/응답 내역

이 모든 정보는 구조화된 데이터로 존재하며, 테스트 실패 시 이 데이터를 AI 에이전트에게 전달하면 에이전트는 마치 사람이 개발자 도구를 사용해 디버깅하는 것처럼 원인을 분석하고 수정안을 제시할 수 있습니다.

## E2E 테스트 구축 전략
▶ [10:17–10:29](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=617s)
테스트를 실제로 어떻게 작성하고 관리하며 운영했는지에 대한 전략입니다.

### 무엇을 테스트할 것인가? (범위 선정)
▶ [10:29–12:04](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=629s)
팀이 작업하는 코드베이스는 6년차, 17만 줄, 220개 페이지에 달하는 대규모 Next.js 프로젝트였습니다. 모든 기능을 테스트하는 것은 비현실적이므로, "가장 중요한 사용자 워크플로우는 무엇인가?"라는 질문에서 시작했습니다.

실패 시 매출에 직접적인 타격을 주거나 사용자의 신뢰를 깨뜨리는 핵심적인 플로우, 즉 **크리티컬 유저 플로우(Critical User Flow)**를 선정했습니다. 예를 들어, 네이버페이 대출 서비스에서는 다음과 같은 플로우가 해당됩니다.
- 사용자가 홈에 진입하여 약관 동의, 본인 확인을 거쳐 결과를 조회하는 플로우
- 신청 가능한 상품 목록과 상생를 확인하고 신청하는 플로우

이러한 핵심 워크플로우를 중심으로 지난 5개월간 37개 라우트에 대한 자동화 테스트를 구축했으며, 안정 브랜치 기준 CI에서 29회 연속 성공하며 안정적으로 운영 중입니다.

### 어떻게 테스트를 작성할 것인가?
▶ [12:04–12:13](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=724s)
#### 1. Playwright 코드젠(Codegen) 활용
▶ [12:13–13:56](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=733s)
팀원 대부분이 Playwright 경험이 없었기 때문에, 첫 단계는 직접 코드를 짜지 않는 것을 목표로 했습니다. Playwright의 공식 도구인 **코드젠(Codegen)**을 사용하면, 사용자가 브라우저에서 직접 클릭하고 입력하는 과정을 기록하여 테스트 코드를 자동으로 생성할 수 있습니다. 생성된 코드는 다소 거칠지만, 이를 기반으로 AI 에이전트의 도움을 받아 기존 코드 컨벤션에 맞게 구조화하는 작업을 진행했습니다.

#### 2. AI 에이전트를 이용한 테스트 코드 자동 생성
▶ [13:56–15:46](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=836s)
Playwright는 공식적으로 테스트 작성 및 유지보수를 위한 3개의 AI 에이전트 시스템을 제공합니다.

```ascii
+-------------------------+
| Planner (테스트 계획)     |
+-------------------------+
↓ (Creates Markdown plan)
+-------------------------+
| Generator (코드 생성)   |
+-------------------------+
↓ (Generates code)
+-------------------------+
| Healer (코드 수정)        |
+-------------------------+
```

1.  **플래너(Planner)**: 코드베이스를 탐색하여 테스트할 플로우를 분석하고, 마크다운 형식의 **테스트 계획서**를 작성합니다. 사람이 코드를 보기 전에 시나리오를 검토할 수 있게 해줍니다.
2.  **제네레이터(Generator)**: 플래너가 만든 계획서를 바탕으로 실제 Playwright **테스트 코드를 작성**합니다. 각 단계를 브라우저에서 실행하며 코드가 정상 동작하는지 확인합니다.
3.  **힐러(Healer)**: 실패한 테스트를 **자동으로 수정**합니다. 실패 지점을 분석하고, 테스트가 통과할 때까지 수정과 재실행을 반복합니다.

#### 3. 에이전트를 위한 하네스 구축
▶ [15:46–19:38](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=946s)
하지만 이 에이전트들을 바로 실행한다고 해서 모든 것이 해결되지는 않습니다. 에이전트는 프로젝트의 도메인 지식, 테스트 전략, 인증 절차 등 고유한 **맥락(Context)**이 없기 때문입니다. 따라서 신입 개발자에게 온보딩을 하듯, 다음과 같은 하네스를 구축해야 했습니다.

1.  **`agent.md`**: 서비스의 핵심 플로우, 인증 방식 등 코드베이스 전체에 대한 안내 문서.
2.  **E2E 테스트 가이드**: 테스트 작성 규칙, 실행 환경, 의사결정 기록 등을 담은 단일 출처 문서.
3.  **유틸/헬퍼 함수**: 인증, 상태 초기화 등 반복되는 로직을 모듈화.
4.  **MCP 서버**: 기획서, 디자인, 이슈, PR 등 사내 컨텍스트에 에이전트가 직접 접근할 수 있는 통로.

이러한 환경을 구축한 후, "Next.js 내비게이션 버그 회귀 테스트 작성해줘"와 같은 간단한 명령만으로 에이전트가 계획서를 만들고(Planner), 검토 후 코드를 생성(Generator)하는 워크플로우가 가능해졌습니다.

## 구축 과정의 주요 결정 사항과 모범 사례
▶ [19:38–19:45](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1178s)
테스트 인프라를 구축하면서 내렸던 주요 결정 세 가지입니다.

### 1. 테스트의 독립성 확보
▶ [19:45–20:47](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1185s)
사용자 상태에 따라 플로우가 분기되는 경우, 모든 테스트가 처음부터 시작하면(예: 약관 동의, 휴대폰 인증) 공통 부분이 반복되어 실행 시간이 길어지고, 공통 부분이 변경되면 모든 테스트가 실패하는 문제가 있습니다.

Playwright 공식 문서에서도 **각 테스트는 서로 독립적이어야 한다**고 권장합니다. 이를 위해, 테스트 시작 전에 **API를 호출하여 특정 상태를 미리 세팅**하는 방식을 채택했습니다. 예를 들어, "휴대폰 인증까지 완료된 상태"를 API로 만들어두고, 테스트는 그 다음 단계부터 검증을 시작합니다.

### 2. 외부 의존성 모킹(Mocking)
▶ [20:47–21:36](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1247s)
네이버 인증서, 홈택스 연동 등 외부 서비스에 의존하는 부분은 테스트의 안정성을 해칠 수 있습니다. 외부 서비스의 작은 변경에도 우리 테스트가 실패할 수 있기 때문입니다.

공식 문서의 "우리가 통제할 수 있는 것만 테스트하라"는 권장 사항에 따라, 호출 위치에 따라 다른 방식으로 모킹을 처리했습니다.
- **브라우저 호출**: Playwright의 `page.route()`를 사용하여 고정된 응답을 반환.
- **서버(SSR, BFF) 호출**: E2E 테스트용 환경 변수를 사용하여 고정 응답을 반환하도록 분기 처리.

### 3. 간헐적 실패(Flaky Test) 예방
▶ [21:36–23:02](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1296s)
간헐적으로 성공과 실패를 반복하는 '플래키 테스트'는 테스트 스위트 전체의 신뢰도를 떨어뜨립니다. 이를 방지하기 위해 작성 단계에서 다음 세 가지 원칙을 적용했습니다.

1.  **하드코딩된 대기 지양**: `await page.waitForTimeout(3000)`과 같은 고정 시간 대기 대신, 특정 조건이 만족될 때까지 기다리는 **조건 기반 대기**(`expect(...).toBeVisible()`)를 사용합니다.
2.  **시맨틱 셀렉터 사용**: `div.some-class`와 같이 깨지기 쉬운 CSS 클래스명이나 DOM 구조 대신, `getByRole('button', { name: '다음' })`과 같은 **시맨틱 셀렉터**를 사용하여 UI 변경에 덜 민감한 테스트를 작성합니다.
3.  **번인(Burn-in) 테스트**: 작성 직후 `repeat-each` 플래그를 사용하여 동일한 테스트를 여러 번 반복 실행합니다. 한 번이라도 실패하면 플래키로 간주하고, PR을 올리기 전에 작성자가 수정합니다.

## AI를 활용한 테스트 유지보수 자동화
▶ [23:02–23:27](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1382s)
테스트 작성뿐만 아니라, 실패한 테스트를 수정하는 것 또한 AI 에이전트에게 맡기는 방법에 대한 내용입니다.

### 힐러(Healer) 에이전트와 트레이스 파일
▶ [23:27–25:04](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1407s)
앞서 소개한 **힐러(Healer) 에이전트**는 실패한 테스트를 진단하고 수정하는 역할을 합니다. Playwright는 테스트 실행 후 모든 과정을 기록한 **트레이스(trace) 파일**을 생성합니다. 이 파일에는 화면 상태, 네트워크 요청, 콘솔 로그 등 디버깅에 필요한 모든 정보가 담겨 있습니다.

힐러 에이전트는 `playwright-trace`와 같은 스킬을 사용하여 이 트레이스 파일을 CLI로 탐색하고, 마치 개발자가 개발자 도구를 보며 디버깅하듯 원인을 분석하고 코드를 수정합니다.

### CI/CD와 연동된 자동 수정 워크플로우
▶ [25:04–26:23](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1504s)
이러한 자동 수정 과정은 CI/CD 파이프라인과 통합될 수 있습니다.
1.  에이전트가 작업물을 PR로 올립니다.
2.  CI에서 E2E 테스트가 실행됩니다.
3.  테스트가 실패하면 머지가 차단되고, 실패 결과(코멘트, 트레이스 파일 아티팩트)가 PR에 남습니다.
4.  에이전트는 이 피드백을 감지하고, CI에서 생성된 트레이스 파일을 다운로드하여 디버깅합니다 (`e2e-debug` 스킬 활용).
5.  힐러 에이전트가 실패한 테스트를 수정하고, 수정된 코드를 다시 PR에 푸시합니다.
6.  CI가 다시 실행되고, 테스트가 통과하면 머지 준비가 완료됩니다.

이 과정을 통해 에이전트가 스스로 작업하고, 검증하며, 실패 시 수정하는 완전한 자동화 사이클이 완성됩니다.

## 자가 개선 루프: 에이전트의 지속 가능한 발전을 위한 회로
▶ [26:23–27:06](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1583s)
에이전트가 피드백을 통해 스스로 작업을 개선해 나가는 것을 **자가 개선 루프(Self-Improving Loop)**라고 부릅니다.

### 가이드와 센서
▶ [27:06–28:56](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1626s)
이 루프가 잘 동작하려면 두 가지 핵심 부품이 필요합니다.

```ascii
+-------------------------+
| Guide (가이드)          |
| - agent.md              |
| - E2E Test (as spec)    |
| - Skills (도구)         |
+-------------------------+
↓ (Provides direction)
+-------------------------+
| AI Agent (에이전트)       |
| (코드 작성 및 수정)       |
+-------------------------+
↓ (Produces output)
+-------------------------+
| Sensor (센서)           |
| - E2E Test (as verifier)|
| - Linter, Type Checker  |
+-------------------------+
↓ (Provides feedback)
+-------------------------+
| Self-Correction (자가 교정) |
+-------------------------+
```

1.  **가이드(Guide)**: 행동하기 전에 방향을 잡아주는 역할 (예: `agent.md`, 스킬).
2.  **센서(Sensor)**: 행동한 결과를 관찰하여 교정을 돕는 역할 (예: 테스터, 린터).

Playwright E2E 테스트는 이 루프에서 **가이드와 센서의 역할을 동시에 수행**합니다. 코드를 읽으면 명세(가이드)가 되고, 실행하면 검증 도구(센서)가 되어 하나의 자원으로 두 가지 역할을 효율적으로 처리합니다.

이는 앤트로픽(Anthropic)이 제시한 **Evaluator-Optimizer 패턴**과 유사합니다. 답을 만드는 LLM(Optimizer)과 그 답을 채점하는 LLM(Evaluator)을 분리하여 피드백 루프를 만드는 개념인데, 이 구조에서 비싸고 느린 LLM Evaluator 대신 **결정론적이고 빠르며 저렴한 Playwright를 센서로 사용**하는 것입니다.

## AI 에이전트 시대, 사람의 역할
▶ [28:56–31:06](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1736s)
에이전트가 개발, 테스트, 수정을 모두 자동화한다면 사람 개발자는 무엇을 해야 할까요?

안드레이 카파시(Andrej Karpathy)는 "사고는 위임할 수 있어도 이해는 위임할 수 없다"고 말했습니다. 에이전트는 CI를 모두 통과시킬 수는 있지만, 그 변경이 프로덕션 환경과 사용자에게 어떤 영향을 미칠지에 대한 깊은 맥락과 책임은 이해하지 못합니다.

따라서 AI 시대에 사람의 역할은 다음과 같이 정리할 수 있습니다.

1.  **자가 개선 루프 설계**: 에이전트가 어떤 워크플로우(예: Planner -> Generator -> Healer) 안에서 일할지, 그 전체적인 흐름과 구조를 설계합니다.
2.  **도구와 맥락 제공**: 코드만 봐서는 알 수 없는 도메인 지식, 테스트 전략 등을 `agent.md`나 가이드 문서로 정리하고, 에이전트가 사용할 스킬과 도구를 제공합니다. 어떤 데이터를 언제 어떻게 제공할지 결정하는 것이 사람의 몫입니다.
3.  **하네스의 진화**: 모델이 발전하면 기존의 하네스 중 일부는 자동화되겠지만, 그만큼 더 어려운 작업을 맡기게 되므로 새로운 형태의 하네스가 필요해집니다. 하네스는 사라지는 것이 아니라 **모양을 바꾸며 모델과 함께 성장**하며, 사람은 계속해서 높은 수준의 도메인 결정을 내려야 합니다.

결국 하네스 엔지니어링은 한 번 만들고 끝나는 것이 아니라, 모델과 도메인의 변화에 맞춰 지속적으로 시스템을 설계하고 발전시키는 일입니다.

## 결론 및 요약
▶ [31:06–32:12](https://www.youtube.com/watch?v=wo0Rsh9hlTo&t=1866s)
발표의 핵심 내용을 세 줄로 요약하면 다음과 같습니다.

1.  테스트 코드는 실행 가능한 **센서**인 동시에, 에이전트에게 방향을 제시하는 **명세(가이드)** 역할을 한다.
2.  이렇게 구축된 테스트 코드는 에이전트가 스스로 코드를 작성하고 개선하는 **자가 개선 루프**의 핵심 부품이 된다.
3.  사람의 역할은 에이전트가 이 루프를 잘 돌 수 있도록 환경을 구축하는 **하네스 엔지니어링**이다.

마지막으로 버셀(Vercel)이 공개한 "에이전트와 책임감 있게 일하기 위한 원칙"을 인용하며 발표를 마무리합니다.
> "에이전트가 자율적으로 움직여도 괜찮은 시스템을 만든다. [...] 원칙은 간단하다. 올바른 행동을 쉽게 할 수 있도록 만들어라."

에이전트가 올바른 행동을 쉽게 할 수 있는 환경을 만드는 것이 바로 우리 개발자들의 새로운 역할임을 강조합니다.
