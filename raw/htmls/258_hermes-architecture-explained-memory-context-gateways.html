The Hermes Agent Loop (3:51–7:52) dig deeper ▶

Hermes' agentic loop is a cyclical process, initiated by user input, involving context building, LLM interaction, tool calling, response, and memory updates.

  • The agentic cycle begins when a user sends a message to the Hermes agent.
  • Hermes dynamically constructs a comprehensive context from its internal memory and predefined prompts.
  • The assembled context and message history are transmitted to the Large Language Model (LLM).
  • The LLM determines whether to invoke available tools (e.g., web search), potentially iterating if multiple are required.
  • Once all necessary tool actions complete, the LLM generates and delivers a final response to the user.
  • The agent analyzes the interaction to extract and store pertinent information for future engagements.

Agent Context Management (7:52–19:58) dig deeper ▶

Hermes manages LLM context using core markdown files and a sophisticated compression strategy to prevent overflow within the context window.

  • The agent constructs context from `soul.md` (personality), `user.md` (user information), and `memory.md` (facts, workflows).
  • Context compression is typically triggered when 50% of the context window is utilized, a customizable threshold.
  • Upon activation, previous conversation messages are summarized and replaced with this summary to optimize token usage.
  • Compression checks occur before each LLM turn and as a fallback mechanism on context window errors.
  • A rich internal prompt guides the summarization process during context compression.

Understanding the Gateway (19:58–28:05) dig deeper ▶

The gateway enables Hermes interaction across various messaging platforms, manages conversation history via SQLite, and handles incoming messages effectively.

  • The gateway enables interaction with Hermes across Telegram, email, Slack, WhatsApp, and SMS messaging platforms.
  • It maintains an `async IO loop` that continuously listens for or polls incoming messages from diverse providers.
  • When a message arrives, it queries a local SQLite database using a session identifier (e.g., `Telegram + session_ID`) to retrieve complete message history.
  • A session manager handles incoming messages while the agent processes a request, allowing interruption, steering, or queuing messages.

Hermes Memory System (28:05–34:43) dig deeper ▶

Hermes' memory includes markdown files, a SQLite database for transcripts, and optional external memory via third-party providers for intelligence.

  • `soul.md`, `memory.md`, and `user.md` are persistent files consistently included in the LLM's context window.
  • This local database stores full transcripts of every interaction session and includes a bare text table for similarity searches.
  • An optional external memory component integrates with third-party providers like MemZero, Super Memory, and Honcho.
  • When enabled, Hermes queries external memory after the first message of a new turn to enhance recall.

Cron Jobs (34:43–39:53) dig deeper ▶

Hermes includes an internal cron system for automated tasks, storing jobs in a JSON file and delivering notifications directly to the user.

  • Hermes incorporates its own internal cron system for automating scheduled tasks, such as sending daily news updates.
  • This system operates independently of server-level cron processes, executing a `tick` function every minute.
  • All scheduled cron jobs are stored in a plain JSON file located at `.hermes/cron/jobs.json`.
  • The results of each job run are saved as a markdown file within a job-specific directory under `.hermes/cron/output/`.
  • Cron job notifications are delivered directly to the user's designated "home" messaging platform by the system itself.
