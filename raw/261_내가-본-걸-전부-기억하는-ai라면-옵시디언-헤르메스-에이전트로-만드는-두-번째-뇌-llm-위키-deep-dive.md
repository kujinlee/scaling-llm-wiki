## AI를 활용한 개인화된 지식 시스템 구축 소개
▶ [0:00–0:52](https://www.youtube.com/watch?v=XT-_Mz18irE&t=0s)
최신 AI 기술은 사용자의 대화 일부를 기억하거나 미리 설정된 규칙에 따라 작동할 수 있지만, 사용자가 의도적으로 수집한 방대한 지식을 체계적으로 기억하고 활용하는 데에는 한계가 있습니다. 이 영상은 옵시디언(Obsidian)과 헤르메스 에이전트(Hermes Agent)를 연동하여, 사용자가 제공하는 자료를 AI가 스스로 읽고 정리하여 개인화된 지식 공간, 즉 '두 번째 뇌'를 구축하는 시스템을 만드는 방법을 소개합니다. 이 시스템은 단순히 정보를 저장하는 것을 넘어, 사용자의 질문에 대해 수집된 자료를 기반으로 맞춤형 답변을 제공하는 것을 목표로 합니다.

## LLM 위키(LLM Wiki) 개념
▶ [0:52–1:09](https://www.youtube.com/watch?v=XT-_Mz18irE&t=52s)
이 시스템의 기반이 되는 아이디어는 테슬라의 AI 디렉터이자 OpenAI 창립 멤버였던 안드레이 카파시(Andrej Karpathy)가 제안한 'LLM 위키'입니다. 이는 거대 언어 모델(LLM)을 사용하여 지식을 정리하고 상호 연결하여 축적하는 방식입니다.

### 기존 AI 자료 활용 방식의 한계
▶ [1:09–1:53](https://www.youtube.com/watch?v=XT-_Mz18irE&t=69s)
일반적으로 AI에게 자료를 주고 질문하는 방식은 다음과 같습니다.
1.  사용자가 다수의 자료 파일을 AI에게 제공합니다.
2.  사용자가 질문을 하면 AI는 실시간으로 관련 자료를 검색하여 답변을 생성합니다.

이 방식은 빠르고 편리하지만, AI가 답변 과정에서 찾은 내용을 별도로 정리하여 남기지 않는다는 단점이 있습니다. 따라서 비슷한 질문을 다시 하면 AI는 매번 처음부터 자료 전체를 다시 검색해야 합니다. 이는 마치 정리되지 않은 서류 더미를 필요할 때마다 뒤지는 것과 같아서, 자료가 아무리 많아도 지식이 체계적으로 축적되지 않습니다.

```ascii
     +--------------------------+
     |   사용자 (User)           |
     +--------------------------+
                  |
                  ↓ (질문 / Asks question)
     +--------------------------+
     |   AI 에이전트 (AI Agent)   |
     |   (매번 자료 전체를 검색)    |
     +--------------------------+
                  |
                  | (답변 / Answers)
     +--------------------------+
     |   정리되지 않은 자료 더미     |
     |   (Unorganized Docs)     |
     +--------------------------+
```

### LLM 위키의 접근 방식
▶ [1:53–2:35](https://www.youtube.com/watch?v=XT-_Mz18irE&t=113s)
LLM 위키는 기존 방식과 정반대의 접근법을 취합니다.
1.  사용자가 새로운 자료를 추가하면, AI가 즉시 해당 자료를 읽고 핵심 내용을 요약하여 구조화된 '노트'로 만듭니다.
2.  기존 노트와 관련된 내용이 있으면 업데이트하고, 서로 링크로 연결합니다.
3.  기존 정보와 상충하는 내용이 발견되면 이를 표시합니다.

이 방식은 일회성 답변을 생성하는 대신, 한 번 정리된 지식을 지속적으로 다듬고 확장해 나가는 방식입니다. 사용자는 자료를 선택하고 좋은 질문을 던지는 역할에 집중하고, 요약, 연결, 정리 등 번거로운 작업은 AI가 전적으로 담당합니다. 결과적으로 자료가 쌓일수록 위키 전체가 더욱 정교하고 유기적인 지식 체계로 발전하게 됩니다.

```ascii
     +--------------------------+
     |   사용자 (User)           |
     +--------------------------+
                  |
                  ↓ (자료 추가 / Adds document)
     +--------------------------+
     |   AI 에이전트 (AI Agent)   |
     | (읽고, 요약, 연결, 정리) |
     +--------------------------+
                  |
                  ↓ (지속적으로 업데이트 / Continuously updates)
     +--------------------------+
     |    구조화된 LLM 위키       |
     |    (Organized LLM Wiki)  |
     +--------------------------+
                  ↑
                  | (질문 시 참조 / Consulted for questions)
     +--------------------------+
     |   사용자 (User)           |
     +--------------------------+
```

## LLM 위키의 구조
▶ [2:35–2:37](https://www.youtube.com/watch?v=XT-_Mz18irE&t=155s)
LLM 위키는 크게 세 개의 층(Layer)과 두 개의 특별한 파일, 그리고 하나의 유지보수 프로세스로 구성됩니다.

### 3개의 핵심 계층
▶ [2:37–3:35](https://www.youtube.com/watch?v=XT-_Mz18irE&t=157s)
LLM 위키의 데이터는 세 개의 논리적 계층으로 나뉩니다.

```ascii
+--------------------------------+
|     스키마층 (Schema Layer)      |
|  (AI 운영 지침서 - agent.md 등)  |
+--------------------------------+
                 |
                 ↓ (Guides operation)
+--------------------------------+
|      위키층 (Wiki Layer)        |
| (AI가 생성/관리하는 정리된 노트) |
+--------------------------------+
                 |
                 ↑ (Reads from)
+--------------------------------+
|    원본 자료층 (Raw Layer)     |
|   (사용자가 수집한 원본 파일)    |
+--------------------------------+
```

1.  **원본 자료층 (Raw Layer)**: 사용자가 수집한 기사, 논문, 이미지, 데이터 파일 등 원본 자료가 저장되는 공간입니다. AI는 이 자료를 읽기만 할 뿐, 절대로 수정하지 않아 원본의 무결성이 보장됩니다.
2.  **위키층 (Wiki Layer)**: AI가 원본 자료를 읽고 생성한 요약, 개념 정리, 비교 분석, 전체 개요 등의 정리된 노트들이 모이는 공간입니다. 이 층의 모든 내용은 AI가 생성, 수정, 연결하며 전적으로 관리합니다.
3.  **스키마층 (Schema Layer)**: AI에게 위키의 운영 방식을 지시하는 '운영 지침서'입니다. 위키의 구조, 페이지 작성 규칙, 자료 처리 절차 등이 `agent.md`나 `claude.md` 같은 파일에 명시됩니다.

### 2개의 길잡이 파일
▶ [3:35–4:07](https://www.youtube.com/watch?v=XT-_Mz18irE&t=215s)
위키 내에는 시스템의 작동을 돕는 두 가지 핵심 파일이 있습니다.
*   **인덱스 파일 (Index File)**: 위키 전체의 목차 역할을 합니다. 모든 페이지의 한 줄 요약과 링크가 정리되어 있어, AI가 질문을 받으면 이 파일을 먼저 참조하여 관련 페이지를 빠르게 찾습니다.
*   **로그 파일 (Log File)**: 위키에서 발생한 모든 활동의 일지입니다. 자료 추가, 질문, 점검 등 모든 이력이 시간 순서대로 기록되어 위키의 성장 과정을 추적하고, AI가 최근 활동 내역을 파악하는 데 도움을 줍니다.

### 정기적인 유지보수: 린트(Lint)
▶ [4:07–4:45](https://www.youtube.com/watch?v=XT-_Mz18irE&t=247s)
위키가 커지면서 발생할 수 있는 정보의 비일관성을 해결하기 위해 '린트(Lint)'라는 건강 검진 프로세스를 주기적으로 실행합니다. AI에게 위키 전체를 검토하도록 지시하여 다음과 같은 항목들을 점검합니다.
*   서로 모순되는 내용
*   오래되어 유효하지 않은 정보
*   다른 어떤 페이지와도 연결되지 않은 고립된 페이지
*   중요하지만 아직 정리되지 않은 개념

이 점검을 통해 위키는 규모가 커져도 일관성과 최신성을 깔끔하게 유지할 수 있습니다.

## LLM 위키 시스템 구축 실습
▶ [4:45–4:54](https://www.youtube.com/watch?v=XT-_Mz18irE&t=285s)
이제 옵시디언과 헤르메스 에이전트를 사용하여 LLM 위키 시스템을 직접 구축하는 과정을 단계별로 설명합니다.

### 1. 사전 준비: 도구 설치
▶ [4:54–5:05](https://www.youtube.com/watch?v=XT-_Mz18irE&t=294s)
#### 옵시디언 (Obsidian) 설치
▶ [5:05–5:40](https://www.youtube.com/watch?v=XT-_Mz18irE&t=305s)
1.  공식 웹사이트(`obsidian.md`)에 접속하여 다운로드 버튼을 클릭합니다.
2.  자신의 운영체제(OS)에 맞는 버전을 다운로드하여 설치합니다.
3.  설치 후, 지식을 저장할 새로운 보관소(Vault)를 생성합니다.

#### 헤르메스 에이전트 (Hermes Agent) 설치
▶ [5:40–7:52](https://www.youtube.com/watch?v=XT-_Mz18irE&t=340s)
1.  이전의 VPS 방식이 아닌, 새로 출시된 데스크톱 앱을 사용합니다.
2.  공식 배포 페이지에 접속하여 자신의 OS에 맞는 버전을 다운로드하여 설치합니다.
3.  설치 후, 모델 프로바이더(Model Provider)를 선택합니다. 이는 헤르메스가 사용할 AI 모델을 어디서 가져올지 결정하는 과정입니다.
    *   **OpenAI OAuth**: 기존 ChatGPT 구독자는 별도 API 비용 없이 사용 가능.
    *   **Anthropic OAuth**: Claude 사용자는 추가 크레딧 필요.
    *   **OpenRouter**: API 키 하나로 여러 모델을 사용할 수 있는 유료 서비스. 영상에서는 오픈라우터(`openrouter.ai`)를 예시로 사용하며, API 키를 발급받아 입력합니다.
4.  설치가 완료되면 헤르메스 에이전트의 워킹 디렉토리(Working Directory)를 앞서 생성한 옵시디언 볼트의 경로로 지정합니다. 이 설정은 헤르메스가 해당 폴더 내에서 모든 작업을 수행하도록 만듭니다.

### 2. 위키 구조 생성 (Scaffolding)
▶ [7:52–9:25](https://www.youtube.com/watch?v=XT-_Mz18irE&t=472s)
헤르메스 에이전트에게 프롬프트를 입력하여 LLM 위키의 기본 폴더 구조와 스키마 파일을 생성하도록 요청합니다. 프롬프트에는 주로 다음과 같은 내용이 포함됩니다.
*   **사용자 정보**: 내가 누구인지, 어떤 목적으로 자료를 수집하는지에 대한 설명.
*   **구조 요청**: LLM 위키의 3계층 구조(Raw, Wiki, Schema)와 핵심 파일(Index, Log, Agent)을 생성해달라는 지시.
*   **참고 자료**: 안드레이 카파시의 LLM 위키 관련 원문 내용.

프롬프트 실행 후 옵시디언에서 확인하면 `raw`, `wiki` 등의 폴더와 `index.md`, `agent.md` 파일이 생성된 것을 볼 수 있습니다. `agent.md` 파일에는 위키의 목적과 운영 규칙이 정리되어 있습니다.

**팁**: 처음 시작할 때는 "회사 업무 전체"와 같이 범위를 너무 넓게 잡기보다, 특정 프로젝트나 주제처럼 작은 범위에서 시작하여 점차 확장하는 것이 효율적입니다.

## 자료 수집 및 개인적 관점 추가
▶ [9:25–9:26](https://www.youtube.com/watch?v=XT-_Mz18irE&t=565s)

### 1. 옵시디언 웹 클리퍼로 자료 수집
▶ [9:26–10:39](https://www.youtube.com/watch?v=XT-_Mz18irE&t=566s)
웹상의 기사나 글을 손쉽게 수집하기 위해 'Obsidian Web Clipper' 크롬 확장 프로그램을 사용합니다.
1.  크롬 웹 스토어에서 'Obsidian Web Clipper'를 검색하여 설치합니다.
2.  설정에서 클립한 노트가 저장될 옵시디언 볼트 이름과 폴더 경로를 지정합니다. LLM 위키 구조에 따라 원본 자료가 저장될 `raw/articles` 폴더로 경로를 설정합니다.
3.  이제 웹 서핑 중 원하는 기사가 있으면 클리퍼 아이콘을 눌러 바로 옵시디언의 지정된 폴더에 마크다운 형식으로 저장할 수 있습니다.

### 2. 개인적 관점 추가를 위한 시스템 개선
▶ [10:39–11:39](https://www.youtube.com/watch?v=XT-_Mz18irE&t=639s)
단순히 자료를 모으고 요약하는 것은 '잘 정리된 자료 모음'에 불과하며, '두 번째 뇌'가 되기에는 부족합니다. 진정한 개인화된 지식 베이스를 만들기 위해서는 사용자의 생각과 관점을 추가해야 합니다.

이를 위해 헤르메스 에이전트에게 다음과 같은 규칙을 추가하도록 요청합니다.
*   자료 요약이 끝난 후, AI가 사용자에게 해당 자료에 대한 짧은 질문을 던질 것.
*   사용자의 답변을 자료와 함께 저장하여, 나중에 '왜 이 자료를 저장했는지'에 대한 맥락을 남길 것.
*   사용자가 바쁠 경우 답변을 건너뛸 수 있는 옵션을 제공할 것.

이 요청을 보내면 헤르메스는 `agent.md` 파일의 지침(instructions) 섹션에 해당 규칙을 자동으로 추가합니다.

### 3. 자료 소화 (Ingest) 및 위키 생성
▶ [11:39–13:07](https://www.youtube.com/watch?v=XT-_Mz18irE&t=699s)
1.  웹 클리퍼를 이용해 `raw/articles` 폴더에 여러 개의 기사를 수집합니다.
2.  헤르메스 에이전트에게 자료를 처리하라는 '인제스트(ingest)' 명령을 내립니다.
3.  헤르메스는 기사를 하나씩 처리한 후, 사전에 설정한 규칙에 따라 사용자에게 "이 기사를 통해 어떤 인사이트를 얻었나요?"와 같은 질문을 던집니다.
4.  사용자가 자신의 생각이나 관점을 답변으로 입력합니다.
5.  모든 처리가 완료되면 헤르메스는 작업 결과를 보고합니다. 옵시디언에서 확인해보면, 원본 기사들을 바탕으로 AI가 'AI 코딩 도구 비교 분석', '기술 개념 정리' 등 다양한 위키 페이지를 생성하고, 관련 페이지들을 서로 링크로 연결한 것을 확인할 수 있습니다. 그래프 뷰를 통해 노트 간의 연결 관계를 시각적으로 파악할 수도 있습니다.

## LLM 위키 활용하기
▶ [13:07–13:30](https://www.youtube.com/watch?v=XT-_Mz18irE&t=787s)
자료가 축적된 위키의 진정한 가치는 '좋은 질문'을 통해 드러납니다. AI가 아닌 사용자의 역할이 가장 중요한 단계입니다.

### 좋은 질문의 조건
▶ [13:30–14:18](https://www.youtube.com/watch?v=XT-_Mz18irE&t=810s)
1.  **위키에 특화된 질문하기**: "커서(Cursor)가 뭐야?"처럼 일반 챗봇도 답할 수 있는 질문 대신, "내 위키에 저장된 자료를 바탕으로, 커서를 추천하는 글과 비추천하는 글의 논리가 어떻게 다른지 비교해 줘."와 같이 내가 모은 자료에 기반한 질문을 해야 합니다.
2.  **범위와 형태를 구체적으로 지정하기**: "AI 코딩 도구 정리해 줘."보다 "내 위키 기준으로, 자율형/IDE/터미널 세 종류로 나눠서 표로 만들어 줘."처럼 원하는 결과물의 맥락과 형식을 명확히 제시하면 훨씬 정교한 답변을 얻을 수 있습니다.
3.  **연결하고 비교하는 질문하기**: 단순 정보 검색을 넘어, "내가 모은 자료들 사이에 서로 충돌하는 주장이 있는지 찾아줘." 또는 "지난번 자료와 비교해서 최신 기술 트렌드가 어떻게 바뀌었어?"와 같이 정보들을 연결하고 비교하는 질문이 위키의 잠재력을 최대한 끌어냅니다.

### 개인적 관점을 재료로 활용하기
▶ [14:18–15:10](https://www.youtube.com/watch?v=XT-_Mz18irE&t=858s)
자료를 인제스트할 때 저장해둔 개인의 관점은 훌륭한 재활용 재료가 됩니다. 예를 들어, 다음과 같이 질문할 수 있습니다.
> "AI 코딩 도구 주제에 대해 내가 남겼던 생각들을 모두 모아서, 유튜브 영상 아이디어 3개만 뽑아 줘."

이 질문에 대해 헤르메스는 사용자의 관점을 요약하고, 이를 바탕으로 구체적인 영상 후보와 우선순위까지 제안하는 새로운 노트를 생성합니다. 중요한 점은 이 답변 자체가 새로운 노트로 위키에 저장된다는 것입니다. 즉, 오늘의 질문에 대한 답변이 내일의 질문을 위한 새로운 자료가 되어 위키가 스스로 성장하고 똑똑해지는 선순환 구조가 만들어집니다.

## 결론: 지속적으로 성장하는 두 번째 뇌
▶ [15:10–15:58](https://www.youtube.com/watch?v=XT-_Mz18irE&t=910s)
LLM 위키는 한 번 만들고 끝나는 시스템이 아니라, **[자료 수집 → AI의 정리 및 연결 → 산출물 생성]** 의 흐름이 계속해서 순환하는 유기적인 시스템입니다. 시간이 지나면서 정보가 낡거나 서로 맞지 않는 내용이 생길 수 있으므로, 주기적인 '린트(건강 검진)'를 통해 위키의 일관성을 유지하는 것이 중요합니다.

이러한 과정을 통해 LLM 위키는 시간이 지남에 따라 사용자의 지식과 관점을 반영하여 점점 더 똑똑해지는 진정한 의미의 '두 번째 뇌'로 발전하게 됩니다.

### 도구에 얽매이지 않는 개념
▶ [15:58–16:38](https://www.youtube.com/watch?v=XT-_Mz18irE&t=958s)
이 영상에서는 옵시디언과 헤르메스 에이전트를 사용했지만, LLM 위키는 특정 도구에 종속된 개념이 아닙니다. 본질은 **'폴더 안의 마크다운 파일들'**과 **'AI에게 운영 방법을 알려주는 규칙 파일'**만 있으면 구현 가능합니다. 따라서 파일을 읽고 쓸 수 있으며 규칙을 따를 수 있는 에이전트(예: Claude, GPT 등)라면 어떤 도구를 사용하든 자신만의 LLM 위키를 구축할 수 있습니다. 가장 손에 익은 도구로 시작하는 것이 좋습니다.
