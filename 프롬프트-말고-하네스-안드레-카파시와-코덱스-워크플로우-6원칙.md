# 프롬프트 말고 하네스 — 안드레 카파시와 코덱스 워크플로우 6원칙

**Channel:** 바이브랩스 | **Duration:** 6:04 | **URL:** https://www.youtube.com/watch?v=Yt67kU3cZfQ

> [!summary] Quick Reference
> **TL;DR:** This video reveals how Andrej Karpathy's 6 principles for LLMs leverage a "harness" workflow, dramatically improving AI coding performance beyond simple prompting.
>
> **Key Takeaways:**
> - Plan tasks and verify code rigorously using features like `plan` and `review` before deployment.
> - Prioritize code simplicity and use context compression (`compact`) to avoid overwhelming the AI model.
> - Define clear goals, success criteria, and verification methods for AI agents, avoiding vague instructions.
> - Design a robust AI "harness" workflow with defined rules, not just isolated prompts, for superior performance.
> - Utilize parallel sub-agents for specialized tasks like debugging or testing to automate repetitive development work.
>
> **Concepts:** ai engineering · coding workflow · openai codex · large language models · agent systems · software development · karpathy principles

## 1. 프롬프팅에서 에이전트 시스템으로의 전환
안드레 카파시의 '클로드 MD' 카드는 코딩 워크플로우 6가지 원칙과 엔지니어 마인드셋을 한 장에 담아냈습니다. 이 카드의 핵심은 LLM이 코딩을 잘한다는 것이 아니라, LLM을 규율 있는 워크플로우에 강제하면 성능이 극적으로 향상된다는 점입니다. 현재 최고의 AI 엔지니어들은 더 이상 단순한 프롬프팅을 넘어 에이전트 주변에 자신만의 시스템(하네스)을 구축하고 있습니다. 클로드 MD나 에이전트 MD와 같은 파일들은 단순한 프롬프트가 아닌 에이전트의 운영 체제처럼 작동하며, 이러한 접근 방식이 코덱스의 변화를 이해하는 데 핵심적인 배경이 됩니다.

---

## 2. 코덱스의 진화: 도구에서 작업장(하네스)으로
오픈AI의 코덱스는 매주 300만 명 이상이 사용하는 거대한 도구로 성장했습니다. 이제 코덱스는 터미널(Codex CLI), macOS/Windows(Codex FS), IDE 확장(Cursor), 클라우드(Codex Web), 모바일 등 다양한 환경에 통합되어 있습니다. 하지만 코덱스를 봐야 하는 진짜 이유는 단순히 코드를 잘 짜는 능력이 아니라, 코덱스가 점차 도구에서 에이전트의 작업장, 즉 '하네스(harness)' 환경으로 변화하고 있기 때문입니다. 이는 AI가 단지 코드를 생성하는 것을 넘어, 개발 워크플로우 전체를 조율하는 오케스트레이션 역할을 수행하게 됨을 의미합니다.

---

## 3. 원칙 1, 2: 코딩 전 사고와 검증의 중요성
카파시 카드의 첫 번째 원칙인 "코딩 전에 생각하기"는 코덱스 CLI의 `plan` 기능으로 구현됩니다. 이는 큰 작업을 단계로 나누고 모호함을 줄이는 데 도움을 줍니다. 또한 `Agent MD` 파일은 BMPM 사용, JS 변경 시 MPM 테스트 자동 실행, 허락 없는 의존성 추가 금지 등 프로젝트 규칙을 저장소에 명시하여 반복적인 채팅 입력 없이 자동으로 작동하게 합니다. 두 번째 원칙인 "멈춰서 검증하기"는 코덱스의 `review` 기능(브랜치, 커밋, 변경 등 4가지 모드)과 `diff status`, 그리고 API 키 누출이나 위험한 명령 실행을 자동으로 잡아내는 `hooks`를 통해 자신의 코드를 꼼꼼히 검토하고 검증하도록 유도합니다.

---

## 4. 원칙 3, 4: 단순함 추구와 필요한 부분만 수정하기
세 번째 원칙인 "단순함이 우선"은 천 줄보다 백 줄의 코드를 선호하며, 더 간단한 방법이 없는지 항상 질문하도록 합니다. 코덱스의 `compact` 기능은 길어진 대화를 의미 손실 없이 압축하여, 모델이 자기 혼란을 끌고 다니는 것을 방지하는 역할을 합니다. 네 번째 원칙인 "필요한 곳만 건드리기"는 관련 없는 코드를 다시 작성하지 않도록 합니다. 코덱스 앱의 `worktree` 기능은 각 에이전트가 격리된 작업 사본에서 일하게 함으로써, 한 에이전트가 다른 작업에 영향을 미치거나 건드릴 일을 원천적으로 차단합니다.

---

## 5. 원칙 5, 6: 목표 중심 실행과 서브 에이전트 병렬화
카파시 카드의 핵심이자 코덱스의 심장인 "목표 중심 실행" 원칙은 `Goal mode`를 통해 구현됩니다. 이 모드는 '이 프로젝트를 완성해 줘'와 같은 모호한 지시 대신, 목표, 범위, 종료 조건, 검증 방법(예: 로그인 접근성 개선, 디자인 유지, 특정 테스트 통과)을 명확하게 정의하도록 합니다. 이는 모델에게 '무엇을 하라'고 지시하는 대신, '성공 기준을 두고 루프를 돌게 내버려 두라'는 카파시의 철학을 반영합니다. 마지막으로 "서브 에이전트로 병렬화" 원칙은 코덱스의 서브 에이전트와 워크트리 자동화를 통해 지원됩니다. 조사, 디버깅, 테스트, 최적화, 검증 등 각기 다른 역할을 하는 에이전트들을 병렬적으로 활용하여 일일 이슈 트리아지, CI 실패 요약 같은 반복 업무를 자동화합니다. 이는 단순한 AI 지원을 넘어선 '오케스트레이션' 개념에 해당합니다.

---

## Conclusion
코덱스의 진정한 잠재력은 단순한 프롬프팅을 넘어, 견고하게 설계된 워크플로우(하네스)에 있습니다. 카파시의 6가지 원칙을 코덱스에 적용함으로써, 모델이 가지는 모호함, 과도한 설계, 혼란 은폐 경향을 크게 줄일 수 있습니다. AI에게 "어떤 프롬프트를 던질까"가 아닌 "AI가 일할 작업장을 어떻게 설계하고 성공 기준을 정의하며 실패 시 대처할까"를 고민해야 하는 시점입니다. 비록 토큰 낭비와 같은 과제도 있지만, 명확한 목표, 검증 명령, 금지 사항, 짧은 체크포인트를 통해 효율적인 AI 기반 개발 환경을 구축하는 것이 중요합니다. 코덱스의 승부처는 프롬프트가 아닌 바로 이 '하네스'에 달려있습니다.
