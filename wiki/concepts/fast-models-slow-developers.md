# Fast Models, Slow Developers

Fast models, slow developers is the principle that a step-change in AI code-generation *speed* does not automatically produce better software — it multiplies whatever habits the developer already has, good or bad. When a model jumps from ~40–60 tokens/second to ~1,200 tokens/second (the corpus cites a roughly 20x leap with OpenAI's Codex Spark), the same "bad habits" that produced mediocre code slowly — massive one-shot prompts, huge unreviewed commits, fleets of unverified agents — now produce *worse* code, and far more of it, generating technical debt at an unprecedented rate. The corrective is counterintuitive: the developer must deliberately *slow down* and reinvest the speed dividend into discipline. Because each individual generation is now nearly free, the time that used to be spent waiting can instead be spent validating, generating and selecting among variations, refactoring continuously, steering the model in real time, and managing context tightly — converting raw speed into quality rather than into volume.

## Key Mechanics

- **Speed multiplies habits, not judgment**: faster inference does not make the *code* better; it makes the developer's existing workflow run faster in both directions. Sloppy practice at 50 tok/s becomes sloppy practice at 1,200 tok/s — the same `[[vibe-coding-vs-agentic-engineering|floor-raising]]` hazard, now accelerated, where fast generation is mistaken for finished work.
- **The speed dividend is reinvested, not banked**: the time formerly lost to slow generation is redirected into quality steps that are now effectively free — comprehensive `[[free-validation-at-speed|validation]]` at every step, `[[generative-variation-selection|generating many variations and cherry-picking]]` the best, and automated refactoring after each task (deleting unused imports, cleaning dead lines, standardizing structure). The dividend pays for discipline rather than for more unverified output.
- **Model orchestration over single-model delegation**: a larger, more intelligent model (e.g. GPT-5.4-class) handles high-level planning and long-horizon reasoning, while a fast model (Codex Spark) acts as the *executor* of individual steps — the strong model can even capture a successful workflow as a reusable "skill" that fast agents then run repeatedly. This is the speed-era expression of `[[per-node-model-routing]]`, splitting planning from execution across model tiers.
- **Active collaboration replaces fire-and-forget**: rather than delegating a whole task to an unattended agent and waiting, the developer sits with the model as a `[[realtime-ai-pair-collaboration|real-time peer]]`, steering decisions and imposing constraints live — because at high speed an unsupervised agent reaches a wrong or bloated state before anyone looks.
- **Tighter context discipline**: faster generation means context windows fill ("compact") far sooner — the corpus cites overflow in ~30 seconds where it once took ~10 minutes — so tasks must be broken into smaller bounded goals and backed by a `[[structured-task-ledger|structured external memory]]` so sessions stay well under context capacity and resume cleanly.

## How It Appears in the Corpus

The Sarah Chieng (Cerebras, "AI Engineer" channel) talk "Fast Models Need Slow Developers" frames the arrival of hyper-fast inference (Codex Spark at ~1,200 tokens/second, ~20x prior state-of-the-art) as a paradigm shift that demands a new developer playbook. It warns that existing habits — one-shot mega-prompts, huge commits, swarms of unverified agents — now generate technical debt at 1,200 tok/s of "even worse code," and prescribes a reinvestment of the freed time into model orchestration (plan with a big model, execute with a fast one, capture workflows as skills), continuous validation, cherry-picking among many generated variations, real-time peer collaboration, proactive steering (banning file deletions, capping diff sizes, limiting to read/write), automated refactoring at every step, and disciplined context management via a four-file external memory system.

## Tensions & Tradeoffs

- **It is the speed-era restatement of `[[friction-as-judgment]]`**: that concept observes the friction of hand-writing code silently carried human review; faster *AI* generation removes even the friction agents used to impose, so the developer must deliberately re-insert slowness where judgment belongs — "slow developers" is the human-pacing counterpart to mechanically rebuilding the quality signal.
- **In tension with `[[token-maxing]]` and parallel-agent maximalism**: the corpus elsewhere celebrates burning tokens and running fleets of `[[parallel-isolated-agents]]`; this talk cautions that at extreme speed, undirected scale-out (many agents, many terminals) is exactly what mass-produces debt. The reconciliation is that speed should be spent on *quality loops* (validation, variation, refactoring) rather than on raw unverified volume.
- **"Slow" is a discipline, not a literal pace**: the prescription is not to throttle the model but to reintroduce human attention, bounded tasks, and verification cadence — so a fast model paired with a disciplined developer is *both* fast and reliable, while a fast model with old habits is merely fast at being wrong.
- **Vantage caveat**: the framing comes from a hardware-vendor talk whose speed figures (1,200 tok/s, 20x) and the named tools are point-in-time, so the durable idea is *reinvest the speed dividend into discipline*, not the specific numbers or products.
