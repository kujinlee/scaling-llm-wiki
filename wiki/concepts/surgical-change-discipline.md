# Surgical Change Discipline

Surgical change discipline is a set of coding principles for restraining an AI agent's tendency to over-reach: *think before you code*, *build the simplest thing*, *make surgical (minimal, targeted) changes*, and *don't improve what wasn't asked* — backed by explicit, checkable success criteria for any autonomous loop. It is a corrective aimed squarely at the LLM coding failure modes the corpus traces to `[[jagged-intelligence]]`: acting confidently on wrong assumptions, papering over hidden confusion, overcomplicating a simple task, and silently expanding scope. Where most methodology concepts structure *what order* work happens in, this one constrains *how much* the agent changes and *how far* it ventures beyond the literal request.

## Key Mechanics

- **Four principles**: think before coding (understand the task before editing, clarifying requirements rather than assuming), build the simplest thing that satisfies it (a senior engineer should not find the code overcomplicated), make surgical changes (touch only what the change requires, avoiding unrelated alterations and side effects), and don't improve what wasn't asked (no unsolicited refactors or scope expansion).
- **Targets named flaws**: the principles are framed as direct antidotes to recurring LLM coding mistakes — wrong assumptions, hidden confusion, and overcomplication — rather than as generic style advice.
- **Explicit success criteria for agent loops**: an autonomous agent must be given a clear, checkable definition of "done" (sometimes named as a distinct "goal-driven execution" principle), so iteration has a termination condition rather than drifting — the same verifiable-goal requirement behind `[[hill-climbing]]` and `[[verifiability-law]]`.
- **Lives in the always-loaded tier**: in the corpus it is delivered as a single-file `[[claude-md]]` plugin appended to an existing context file, so the discipline is re-asserted every session rather than depending on the operator remembering to prompt for it. Embedding the principles into `claude.md` makes them part of the model's inherent "personality" — constant, always-on behavioral guardrails rather than a per-task invocation, the embedded-constraint pole of `[[embedded-guardrails-vs-triggered-skills]]`.

## How It Appears in the Corpus

The Nuno Tavares "6 Claude Code GitHub Repos" overview presents "Karpathy Skills" — based on Andrej Karpathy's observations about common LLM coding flaws — as four core principles ("think before you code," "build the simplest thing," "make surgical changes," "don't improve what wasn't asked") plus an emphasis on defining clear success criteria for agent loops. It is described as a simple single-file Claude MD plugin that rectifies wrong assumptions, hidden confusion, and overcomplication, guiding the agent toward more robust, targeted code, and can be applied to enhance an existing Claude MD file.

The Eric Tech video "Karpathy's Skill Just Fixed Claude Code's Biggest Problem" corroborates and sharpens the same four principles, naming them Think Before Coding, Simplicity First, Surgical Changes, and Goal-Driven Executions, and ties each to a specific LLM failure mode Karpathy identified (assuming instead of clarifying, writing 1,000 lines for a 100-line task, and introducing unintended changes without understanding side effects). It frames the principles as guardrails embedded directly into the `claude.md` file so they become part of the LLM's "personality" — installable globally via the plugin marketplace or per-project via a `curl` command, and recommends using Claude Code itself to resolve conflicts (duplicate H1 tags, irrelevant meta-framing) when merging them into an existing context file. Distinctively, it argues that principles 2 (Simplicity First) and 3 (Surgical Changes) are *unique* guardrails not explicitly covered by triggered-skill frameworks like Superpower or G-stack — those teach careful work, but Karpathy's principles specify *how much* to do and *where* — while principles 1 and 4 overlap conceptually with those frameworks' planning phases. It closes by recommending a hybrid: keep the four principles as constant embedded guardrails and additionally trigger task-specific skills (brainstorming, debugging, planning, simplification, TDD) as each task demands (`[[embedded-guardrails-vs-triggered-skills]]`).

## Tensions & Tradeoffs

- **Direct tension with `[[aspirational-prompting]]`**: that pattern instructs the agent to envision the most ambitious "10x" version of a solution, while surgical change discipline says build the simplest thing and don't improve what wasn't asked. The reconciliation is altitude-dependent — aim high when *scoping and planning* (ambition belongs in the spec), stay surgical when *implementing* (restraint belongs in the diff).
- **Clear success criteria is the verifiability ingredient**: defining "done" is exactly what `[[verifiability-law]]` and `[[self-verification-loop]]` require for trustworthy autonomy, so this principle is the entry condition that keeps an agent loop from running without a stopping rule.
- **Risk of under-building**: read too literally, "don't improve what wasn't asked" can suppress a legitimately needed refactor or harden technical debt, so the discipline still presupposes human `[[engineering-taste]]` to judge when a broader change is actually warranted.
- **Prose principles, not enforcement**: like all of `[[claude-md]]`, these are requests the model is asked to honor, weaker than the mechanical block of an `[[auto-correction-loop]]` or `[[permission-tiering]]` — they shape behavior but cannot guarantee it. The corpus also pairs the discipline with the upstream `[[plan-first-workflow]]` and `[[thinking-vs-understanding]]` move of comprehending the task before touching code.
- **Embedded constant vs. triggered procedure**: because the principles are constant behavioral constraints rather than a per-task workflow, the corpus positions them as the *embedded guardrail* layer that complements — not replaces — *triggered* procedural skills (`[[procedural-knowledge-skills]]`, `[[spec-driven-development]]`, `[[agent-tdd]]`); the recommended setup runs both at once, the synthesis named in `[[embedded-guardrails-vs-triggered-skills]]`. Whether a behavior belongs in the always-loaded tier (and so costs standing context budget, the `[[context-rot]]` risk) or behind an on-demand trigger is the design judgment that distinction makes explicit.
