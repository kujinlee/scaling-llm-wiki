# Ubiquitous Language for Agents

Ubiquitous language for agents is the practice of borrowing Domain-Driven Design's "ubiquitous language" — a single shared vocabulary spanning domain experts, developers, and the codebase — and persisting it as a living document the agent reads and updates, so that human and AI name every domain concept identically. The motivating problem is that an open-ended interview/"grilling" elicitation loop ([[collaborative-prompt-elicitation]]) repeatedly re-derives the same project jargon from scratch: the operator must re-explain domain terms ("standalone video," "pitch") and non-obvious codebase concepts every session, and the agent's replies stay verbose because nothing pins down the agreed meaning. The fix is to formalize the language up front into a context.md glossary — and to record hard-to-reverse design choices as Architectural Decision Records (ADRs) — so the shared understanding produced during elicitation is captured once and reused, rather than evaporating when the session ends. It is the documentation-side complement to elicitation: the interview discovers the language, the glossary makes it durable.

## Key Mechanics

- **context.md as a living glossary**: a markdown file holds the project's shared terminology — each domain term and its agreed definition, plus the relationships between terms (e.g. how a "pitch" relates to a "standalone video") — so the agent and the human draw on one canonical vocabulary instead of improvising synonyms. It is a standing [[context-engineering]] artifact distinct from [[claude-md]]'s build/test/convention knowledge: where claude-md carries how to operate the codebase, the glossary carries what the domain's words mean.
- **Architectural Decision Records for the non-glossary residue**: choices that are significant and hard to reverse but do not reduce to a term definition — why a structural approach was taken — are written as small standalone ADR markdown files, capturing the why behind design decisions the glossary cannot hold. This is the design-decision counterpart to [[reasoning-trace-memory]], persisting decision provenance as documentation rather than as queryable agent memory.
- **Grilling against the glossary, not from zero**: an elicitation skill ("Grill with Docs") is instructed to first look for an existing context.md to bootstrap the shared language, challenge the user's word usage against that glossary, sharpen fuzzy language by discussing concrete scenarios, cross-reference terms with the actual code, and resolve terminology collisions — prioritizing language alignment before descending into implementation detail.
- **Dynamic update during the session**: as new terms and definitions are agreed mid-conversation, the skill writes them back into context.md, so the glossary grows as a byproduct of the work rather than as a separate documentation chore — the same capture-once-reuse-forever loop as [[compound-engineering]], applied to vocabulary.
- **Language alignment shapes the code**: because variable names, file names, and overall structure are made to match the documented ubiquitous language, the resulting codebase becomes easier to navigate — the glossary is not just an agent prompt but a naming contract the implementation honors.

## How It Appears in the Corpus

The Matt Pocock video "I stopped using /grill-me for coding" recounts evolving a praised "Grill Me" elicitation skill into "Grill with Docs" after hitting two limits: Grill Me produced verbose AI responses and forced constant re-explanation of domain jargon and non-obvious codebase concepts, with no persistent record of the refined language. The fix adopts Domain-Driven Design's Ubiquitous Language as a "Ubiquitous Language skill" that maintains a context.md glossary, plus Architectural Decision Records (simple markdown files) for non-obvious, hard-to-reverse design decisions that do not fit the glossary. "Grill with Docs" combines the grilling loop with this documentation: it bootstraps from an existing context.md, challenges language against the glossary, sharpens fuzzy terms through concrete scenarios, cross-references with code, and saves agreed definitions back to the file. A live demo shows it resolving a terminology collision and defining the "pitch"/"standalone video" relationship before any implementation. The reported payoff is markedly more concise AI replies and easier-to-navigate code, and the video is explicit that the original "Grill Me" remains better for codeless use cases (personal writing, early ideation) while "Grill with Docs" is for an existing or nascent codebase.

## Tensions & Tradeoffs

- **Complement to, not replacement for, [[collaborative-prompt-elicitation]]**: the grilling interview is still how the language is discovered; the glossary is how it is retained. The corpus is explicit that the documentation-free elicitation skill stays preferable where there is no codebase to anchor terms (personal writing, idea-stage projects), so the two are phase-specific — elicitation alone for codeless work, elicitation-plus-glossary once code exists.
- **Glossary quality bounds the alignment**: a context.md only helps if its definitions are accurate and actually honored in naming — a stale, vague, or contradicted glossary silently feeds the agent the wrong meaning, the same "quality of the artifact bounds the trust" caveat that bounds [[claude-md]], and an unmaintained glossary drifts exactly as a synthesized wiki does.
- **Standing context cost**: every term loaded into the always-read glossary consumes window budget, re-raising the [[context-rot]] dilution risk — so the glossary must stay a concise, high-signal vocabulary rather than an exhaustive dictionary, the same map-not-manual discipline that bounds claude-md.
- **Two artifacts, two maintenance burdens**: splitting durable knowledge into a glossary (context.md) and decision records (ADRs) cleanly separates "what words mean" from "why we chose this," but it is two documents to keep current as the domain and design evolve — and a wrong ADR confidently records a justification for a decision that may no longer hold.
- **Established human practice ported to AI**: the durable insight is that Domain-Driven Design's decades-old human-coordination techniques (ubiquitous language, ADRs) translate directly to human-agent coordination — but the source is a single practitioner's skill demonstration, so the concision and navigability gains are illustrative of the pattern (formalize shared vocabulary as living docs) rather than measured outcomes.
