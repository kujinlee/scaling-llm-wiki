# Harness Engineering

Harness engineering is the discipline of rigorously pre-defining the environment, rules, and reference material that constrain a powerful AI model so it acts in accordance with the operator's intent. A capable model is treated like a powerful but undirected engine — or a wild horse — and the harness (code scripts, policy/specification documents, and reference files) is the tack that channels that raw capability into reliable execution of complex, real-world tasks. Crucially, harness engineering does not try to control the *model*; it designs the *external environment and rules* the model operates within. It is framed as the third stage in an evolution of AI-coding practice: from prompt engineering (optimizing a single LLM output), through `[[context-engineering]]` (curating the perfect context for a single agent), to harness engineering (chaining multiple agent sessions together and structuring the environment around them).

## Key Mechanics

- The harness is assembled from three kinds of artifact: executable code/scripts, written rules and specifications, and reference files the agent can consult.
- It is defined *up front*: the operator specifies how the model should behave before the agent runs, rather than steering each step interactively.
- Harness quality, not raw model strength, is the limiting factor — a well-harnessed agent handles complex work stably, while a weak harness makes even a strong model unreliable. The corollary is a diagnostic rule: when an agent underperforms, inspect the harness before blaming the model.
- A harness strings *multiple* coding-agent sessions together, integrating validation, targeted context curation, and chaining; this is what lifts output past the reliability ceiling of a single prompt or single agent.
- **Fix the harness, not the prompt.** The defining philosophy is that when an agent makes a mistake, the response is not to patch the wording of a prompt but to improve the *structure* so that the mistake becomes structurally impossible to repeat. A prompt is a *request*; a harness is *enforcement*.
- It is motivated by two recurring failure modes: `[[context-decay]]` (a finite context window means the agent forgets earlier instructions or terminates work early as a task runs long) and the *rules-and-fence problem* (the agent knows all the relevant information yet, lacking structural constraints, takes unintended or dangerous actions — e.g. deleting a database while building a payment flow). The second is a problem of missing control structure, not missing information.
- The harness is often decomposed into three pillars: a **context file** (`[[claude-md]]` / `agent.md`) read first every session to re-ground the agent; an **automatic enforcement system** of linters and pre-commit hooks that mechanically block rule violations and feed errors back for self-correction (the `[[auto-correction-loop]]`); and **garbage collection** (`[[ai-garbage-collection]]`) that periodically prunes the bad code patterns the agent accumulates.
- `[[claude-md]]` is a concrete harness artifact: project documentation is one layer of the rules-and-reference material that constitutes the harness. A `[[deterministic-workflow-orchestration]]` definition is another, more executable layer.

## How It Appears in the Corpus

The Korean "AI agent transition" talk frames harness engineering as the single most important factor in successfully deploying agents, claiming a well-harnessed agent approaches AGI-level task transfer. The speaker runs two harness layers — Claude Code for professional work and a lighter everyday-assistant agent — on an always-on Mac mini controlled over Telegram.

The Cole Medin / Archon tutorial positions harness engineering as "the next evolution of AI coding" and quantifies its payoff: a well-designed harness is cited as raising pull-request acceptance rates from under 7% to nearly 70% by combining validation, specific context curation, and the chaining of agent sessions. Archon is presented as an open-source "harness builder" that encodes a development process as reusable workflows (see `[[deterministic-workflow-orchestration]]`), and enterprise harness use (Stripe, Anthropic's internal tooling) is offered as evidence of the approach's importance.

The Garry Tan / Y Combinator "Tokenmaxxing" talk advances a contrasting design stance — keeping the harness deliberately *thin* and generic while pushing behavior into thick natural-language skills (`[[thin-harness-fat-skills]]`), backed by heavy automated testing — showing that *how much* harness to build is itself contested across the corpus.

The Cole Medin "AI Dark Factory" experiment positions the harness as the rung between human-in-the-loop coding and full autonomy: its coding-autonomy ladder names Level 4 a "harness" that runs unattended overnight (e.g. an agentic loop) with humans reviewing results later, and Level 5 the `[[dark-factory]]` that removes pre-merge review entirely. There the Arkon/Archon harness builder supplies the executable scaffold while a three-file `[[governance-layer]]` supplies the rules-and-reference constraints — the two halves of a harness this corpus consistently separates. The follow-up "AI Dark Factory is ALIVE" live stream then runs that harness in production and shows its workflows failing and being corrected on air — making "fix the harness, not the prompt" an *operational* discipline applied to the harness's own defects (`[[harness-debugging]]`), not only an authoring-time principle.

The 캐슬 (아는 개발자) explainer traces the term to Mitchell Hashimoto (2023), introduced to stop AI coding agents from repeating the same mistakes, and uses the wild-horse-to-racehorse metaphor to make the point that harness engineering shapes the environment, never the model itself. It crystallizes the "prompt is a request, harness is enforcement" philosophy, the three-pillar structure (context file + automatic enforcement + garbage collection), and a concrete on-ramp: start with an `agent.md` and a pre-commit hook (1–2 hours), then refine the harness incrementally off real agent failures. It also forecasts that as models grow more capable harnesses will *simplify* — eventually toward self-configuring "self-harness engineering" and stack-specific harness templates shipped with frameworks — and reframes the developer's role as a "relocation of rigor" toward designing and supervising the environment in which the agent produces correct code.

## Tensions & Tradeoffs

- Front-loaded effort: the reliability payoff requires substantial up-front investment in scripts, rules, and references before the agent is useful — though the 캐슬 source counsels starting minimal (one context file, one hook) and growing the harness only in response to observed failures, rather than over-engineering it in advance.
- Designed vs. discovered reliability: harness engineering promises reliability *by structure*, but the live dark-factory run shows much of a harness's real reliability is only found by running it and watching its workflows fail — the operational counterpart `[[harness-debugging]]`, where the harness is debugged like any other production software.
- Where the weight belongs: this corpus also contains a *thin-harness* counter-position (`[[thin-harness-fat-skills]]`) arguing the loop should stay generic and the intelligence should live in flexible prose skills plus rigorous testing, not in a heavy deterministic scaffold. Both agree the surrounding artifacts are decisive; they disagree on the source of reliability.
- Self-erasing trajectory: the 캐슬 view that harnesses get *simpler* as models improve sits in tension with the Archon/dark-factory direction, where harnesses grow into elaborate multi-workflow systems — whether harness complexity trends up or down with model capability is unresolved.
- Garbage In, Garbage Out: a harness disciplines *how* an agent works but cannot rescue a vague or wrong underlying plan; for ill-defined ideas the source advises validating the concept and shipping quickly before investing in harness structure.
- Maintenance and information overload: as the number of harnessed agents grows, the corpus notes a rising burden of upkeep and oversight, which must be met with more deliberate delegation and management.
- The headline 7%→70% PR-acceptance figure is a motivating statistic from a tool tutorial, not an independently controlled measurement, and depends heavily on the quality of the validation and context layers the harness wraps.
