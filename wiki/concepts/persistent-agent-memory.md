# Persistent Agent Memory

Persistent agent memory is the architecture by which a coding agent retains facts, decisions, and conversation history across sessions and automatically reloads the relevant pieces into each new context. Rather than one ever-growing file, mature designs stratify memory into tiers — short-term logs versus long-term facts — index it for retrieval, and inject only the relevant slice at the start of (or during) a session, so the agent effectively "remembers" without re-reading everything. The design space decomposes cleanly along three axes — how memory is stored, injected, and recalled (`[[memory-storage-injection-recall]]`).

## Key Mechanics

- Structured tiers: separate files for cross-project facts (a `general` store), per-domain/topic knowledge, and per-tool notes, with an index file (`memory.md`) pointing to the granular documents. Organizing by topic makes recall consistent and automatic.
- Storage substrate: the durable layer can be a `[[filesystem-memory-store]]` — a persistent file system attached as a resource to ephemeral sessions and explored with shell tools — beneath the tiered-markdown organization.
- Hook-driven injection: session-start and user-prompt-submit hooks load or inject the relevant memory automatically, rather than relying on the model to remember to call a search tool — which raises the reliability of recall (see `[[lifecycle-hooks]]`).
- Semantic retrieval: documents are chunked into vector embeddings so queries match on meaning rather than keywords, and the top-k relevant memories are injected per prompt — a RAG approach shared with `[[llm-knowledge-wiki]]`. Hybrid ranking pairs semantic vectors with lexical keyword matching (BM25) so exact terms and conceptual matches are both surfaced.
- Short-term vs long-term with consolidation: daily notes capture recent activity while a long-term store holds durable facts; a "dreaming"/promotion process elevates frequently accessed short-term items into long-term memory (`[[memory-consolidation-dreaming]]`) — an automated analogue of `[[compound-engineering]]`'s knowledge accumulation. A scheduled (e.g. nightly or weekly) consolidation job re-indexes transcripts and prunes/merges files.
- Complete-capture vs. curated-save storage: one strategy captures *every* conversation turn automatically (treating markdown as the source of truth and rebuilding the vector index from it), the other has the agent decide what is "memory worthy" via explicit add/replace/remove tools under character caps — the two storage poles of `[[memory-storage-injection-recall]]`, best combined.
- Frozen-snapshot injection: a rich but bounded context bundle (project file, consolidated facts, user profile, persona) is loaded and *cached per session* as a "frozen snapshot," with mid-session updates written to disk but only loaded in the next session — keeping injection lean and stable within a run.
- Tiered recall: retrieval escalates cheapest-first through `[[progressive-disclosure-retrieval]]` — an in-context check, then hybrid semantic/keyword search, then context expansion, then raw transcript as a last resort — so token-expensive retrieval fires only when needed.
- Memory hygiene: consolidation can also *prune and merge* memory files to keep the long-term store from bloating, not only promote them — a maintenance counterweight to the accumulation that otherwise drifts toward `[[context-rot]]`.
- Verbatim vs summarized storage: some systems store conversations verbatim (lossless, exact word-for-word recall) behind a relational store for entities plus a vector store for searchable chunks, written silently via session-end/pre-compaction hooks; others summarize to save space.

## How It Appears in the Corpus

The Simon Scrapes memory-systems comparison lays out a six-level ladder. Beyond the native files, Level 2 adds structured topic files (`general.md`, `domain/topic.md`, `tools/tool.md`) loaded via hooks; Level 3's MemSearch plugin (inspired by OpenClaude) keeps a long-term `memory.md` plus daily notes and a "dreaming" promotion step, chunking memories into vectors and injecting the top three matches via a user-prompt-submit hook; Level 4's Mem Palace stores conversations verbatim across an SQL entity store and a Chroma vector DB, indexed on session-end/pre-compaction hooks for precise word-for-word recall. The presenter reports personally implementing up to Level 3.

The Nate Herk "Every Level of Claude" overview names a Level-5 "Memory Consolidation" (Autodream) step that prunes and merges memory files automatically — the explicit maintenance counterpart to the "dreaming" promotion process, aimed at keeping the long-term store compact rather than only growing it.

The Simon Scrapes "Master Claude Memory in 23 Minutes" walkthrough designs a hybrid around the three pillars (`[[memory-storage-injection-recall]]`) by combining two open-source systems. *Memarch* captures every conversation turn via a Claude Code stop hook, summarizing each turn with a cheap fast model (Haiku) into date-stamped markdown as its source of truth, then chunking, hashing, and vectorizing into a local Milvus store that is rebuildable from the markdown — and recalls through a three-tier `[[progressive-disclosure-retrieval]]` cascade (semantic/keyword search → expand → raw dialogue), but notably has *no* injection layer of its own. *Hermes* instead curates: the agent decides what to save via add/replace/remove tools under character caps into `memory.md` (environment) and `user.md` (profile), a curator prunes and consolidates weekly, and at session start it injects a cached ~1,300-token "frozen snapshot" (`claude.md` + `memory.md` + `user.md` + `soul.md`), with fast in-context lookup but no semantic search. The recommended hybrid takes Memarch's complete vectorized capture and tiered recall, Hermes' curated frozen-snapshot injection (~3,000 cached tokens), a nightly consolidation index, and a "Tier 0" zero-cost check of memory files already in context before any vector search runs.

The Anthropic "Agents that remember" (Claude channel) workshop grounds the same architecture in a platform primitive: against the failure of isolated Cloud-Managed-Agent sessions, it attaches a persistent `[[filesystem-memory-store]]` as a resource (mounted as a file system the agent explores with `bash`/`grep`, scoped per-user or per-workspace) for immediate cross-session recall, and adds `[[memory-consolidation-dreaming]]` — an asynchronous, non-destructive multi-agent batch that distills, fact-checks, deduplicates, and enriches the store — as the background maintenance layer. It frames sessions, memory stores, and dreaming as composable layers that keep memory manageable as interaction volume scales.

## Tensions & Tradeoffs

- Verbatim vs summarized is a direct tradeoff: verbatim recall loses nothing and supports exact retrieval but stores opaque, non-human-readable data; summarization is compact and legible but lossy.
- Complete-capture vs. curated-save: capturing every turn maximizes recall coverage but spends indexing tokens and risks burying signal in noise, while letting the agent decide what is "memory worthy" stays lean but depends on that judgment being right — the same coverage gap, which the hybrid sidesteps by running both and reconciling at recall time.
- Retrieval quality is the bottleneck (as with `[[llm-knowledge-wiki]]`): hook-injected top-k memories only help if chunking and embeddings surface the right ones, and bad retrieval silently feeds wrong context — itself a path to `[[context-rot]]`.
- Automatic injection vs tool-call: hook injection is more reliable but spends context budget on every prompt; tool-call retrieval (e.g. the Claude Mem alternative) is leaner but depends on the model choosing to search. The frozen-snapshot approach is a middle path — a one-time per-session cached load rather than per-prompt injection.
- Consolidation cost and trust: a thorough background consolidation (`[[memory-consolidation-dreaming]]`) is token-expensive by design and only as good as the harness that runs it — a flawed dream can enrich in the wrong direction, which is why a non-destructive output store and reviewable diff matter before the improved store is adopted.
- Infrastructure cost of the hybrid: covering all three pillars well means standing up a stop hook, a nightly consolidation job, a vector store, *and* snapshot injection logic simultaneously — markedly more moving parts than native memory, the price of not being weak on any single axis.
- Capability vs ownership: richer stores (dashboards, team sharing) often trade away local readability or keep data on a vendor's servers — a portability and custody concern taken up by `[[cross-tool-memory]]`.
