# Agent-Native Knowledge Graph

An agent-native knowledge graph is a personal knowledge base deliberately structured for an *AI* to consume rather than for a *human* to browse — built from small atomic notes, a rich taxonomy of typed nodes, and semantically-typed edges — so an agent can locate and retrieve exactly the relevant fact, decision, or relationship without parsing large documents of irrelevant text. Its motivating claim is that the dominant human-centric PKM systems (the PARA method of "Building a Second Brain," organized as Projects / Areas / Resources / Archives in tools like Obsidian) were designed for human organization and are actively *inefficient* for AI: long notes, untyped links, and thin metadata force the model to read whole documents to answer narrow questions, burning tokens and degrading precision. The fix is to redesign the *structure of the knowledge itself* around how an agent retrieves — the personal-knowledge counterpart to `[[agent-native-infrastructure]]`, which reshapes docs and APIs for an agent reader.

## Key Mechanics

- **Atomic notes**: knowledge is split into small, single-idea notes (the source cites 50–300 lines) so the agent ingests a tightly-scoped unit rather than a sprawling document — the granularity that makes precise, low-cost retrieval possible instead of forcing a full-document read.
- **Typed nodes over human folders**: instead of four human-facing categories, the graph uses a rich taxonomy of node *types* (the source names 16, e.g. Pillars, Decisions, Concepts, Questions, Playbooks, Facts, Sources) so the agent understands *what kind of thing* each note is and can target the right category — surfacing "which decision was made" without scanning prose.
- **Semantic edge types**: links carry meaning rather than being bare pointers — a set of relationship types (the source names 10, e.g. *supports*, *contradicts*, *depends on*, *derived from*, *part of*, *preceded by*, *authored by*) tells the agent *how* two notes relate, so it can traverse the relationship without reading both documents to infer it. This is the relationship-first discipline of `[[graph-augmented-retrieval]]` and `[[context-graph]]` applied to a personal store.
- **Token efficiency as the payoff**: because the agent finds and pulls only the relevant typed node along its typed edges, token consumption for the same query collapses (the source's demonstration: ~9,000 tokens down to ~600) — the inverse lever to `[[token-maxing]]`, spending structure up front to spend tokens sparingly at query time.
- **AI-assisted construction and maintenance**: the agent itself can build and organize the graph (classifying notes into node types, wiring edges), reducing the human effort of maintaining the structure — the same agent-maintained-wiki move as `[[llm-knowledge-wiki]]`.
- **Institutional-knowledge durability**: structured this way, the graph becomes a queryable repository of company or personal knowledge that outlives individual team members and eases onboarding, applicable across data analysis, design, SOP creation, and client knowledge libraries.

## How It Appears in the Corpus

The AI Impact video "Don't Use Karpathy's Second Brain (I Built Something Better)" introduces the "Infinite Brain" system, arguing that generic models lack business context and that the route to "high-quality expert AI" is a personalized knowledge graph engineered for AI rather than humans. It critiques human-centric PKM (PARA / Building a Second Brain in Obsidian) for massive notes, untyped links, poor metadata, and poor scoped retrieval, and proposes three pillars: atomic notes (50–300 lines), 16 specific node types (Pillars, Decisions, Concepts, Questions, Playbooks, Facts, Sources, etc.), and 10 semantic edge types (supports, contradicts, depends on, derived from, part of, preceded by, authored by, etc.). It reports a token reduction from ~9,000 to ~600 for the same data and frames the structure as AI-buildable, context-rich institutional knowledge usable across many domains.

## Tensions & Tradeoffs

- **Sharpens the human-vs-agent reader split with `[[second-brain]]` and `[[llm-knowledge-wiki]]`**: those concepts build an agent-readable wiki but largely inherit a human-browsable markdown layout (Karpathy's `raw`/`wiki`/`index` folders, Obsidian graph view); this concept argues that human-legible structure is itself the inefficiency and that the *schema* must be redesigned for the agent — node types and typed edges over folders and bare links. It is a structural critique of, not a replacement for, the second-brain pattern: both externalize personal knowledge, but they disagree on whether human readability should drive the layout.
- **Structure cost vs. retrieval payoff**: a rich 16-node / 10-edge taxonomy is far more upfront modeling than dropping notes in folders, and the token savings only materialize if notes are actually classified and linked correctly — the same build-cost-vs-precision tradeoff as `[[codebase-knowledge-graph]]` and `[[graph-augmented-retrieval]]`, here borne by the personal-knowledge curator.
- **Schema quality bounds retrieval**: the agent finds only what the node typing and edge semantics expose, so a mis-typed note or a missing *contradicts* edge silently hides relevant context — relationship depth caps how far reasoning can hop, the recurring "retrieval/structure quality is the new bottleneck" caveat of `[[context-graph]]` and `[[llm-knowledge-wiki]]`.
- **A typed `contradicts` edge presupposes maintenance**: encoding contradictions explicitly is powerful, but keeping the graph internally consistent as it grows is exactly the standing-maintenance problem `[[knowledge-base-linting]]` exists to carry — an unmaintained AI-optimized graph rots into stale or conflicting typed claims just as a synthesized wiki does.
- **Vantage caveat**: the structure, the 16/10 type counts, and the 9,000→600 token figure come from a single intermediate tutorial demonstrating one product ("Infinite Brain"), so they are illustrative of the *pattern* — designing knowledge structure for an agent reader — rather than measured outcomes; the durable idea is AI-first knowledge structuring, not the specific taxonomy or numbers.
