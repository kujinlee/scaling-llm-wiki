# Agent-Native Verification

Agent-native verification is the practice of building verification *into the development artifacts themselves* so that an agent can read and confirm correctness directly, rather than treating testing as an external step bolted on after the code is written. The motivating problem is duration: as agents run longer and tackle more complex tasks, they need a way to check their own work that does not depend on a human watching each step. The defining move is to have components *publish their internal state as data contracts in the DOM* — exposing the truth of what the application is actually doing in a form the agent can parse — so verification is grounded in machine-readable state rather than in pixels the agent must interpret or a human must eyeball. It is `[[agent-native-infrastructure]]` applied to the *verification surface*: where that concept reshapes docs and APIs for an agent reader, this reshapes the artifact's testability so an agent is the primary verifier.

## Key Mechanics

- **State published as a DOM data contract**: components write their internal state into the DOM as a structured, readable contract, so an agent reads the *actual* state directly instead of inferring it from rendered appearance — the verification analogue of exposing an agent-legible interface rather than forcing screen-scraping (`[[computer-use-automation]]`).
- **One verification, three runtimes**: the same embedded verification can be exercised three ways — as a *human-readable dashboard*, as an *agent-first browser experience* (driven through Playwright MCP, a `[[model-context-protocol]]` server), or *headlessly in CI* — so the identical correctness check serves a human reviewer, an autonomous agent, and an automated pipeline without rewriting it per audience.
- **Built from familiar primitives, re-arranged**: the corpus's worked example assembles Storybook fixtures, a testing library, and Playwright MCP around a React to-do app — existing tools recomposed so verification lives inside the artifact rather than as a separate test suite the agent might skip.
- **Evidence, not assertion**: the process can record *video clips* of the verification run as durable evidence that the check actually executed — the artifact-level counterpart to the `[[self-verification-loop]]`'s "produce evidence before declaring success," and a persisted trace akin to `[[reasoning-trace-memory]]` for the verification itself.
- **Designed for long-running autonomy**: because the verification is embedded and machine-readable, an agent working unattended over a long horizon can keep confirming its own work against the published state, rather than drifting until a human checks.

## How It Appears in the Corpus

The Anthropic "How we Claude Code" workshop (Ara, Applied AI team) devotes its second half to agent-native verification, arguing that for long-running agents verification must be embedded directly into the artifacts. It demonstrates a framework using Storybook fixtures, a testing library, and Playwright MCP on a React to-do app, where components publish their internal state as data contracts in the DOM so agents can read and verify those states directly. The same verification is shown running as a human-readable dashboard, an agent-first browser experience, and headlessly in CI, and the process records video clips as evidence — a practice the speaker notes is already used by Anthropic's own Claude Code team. The talk frames the approach as re-arranging familiar primitives to make development artifacts inherently agent-first.

## Tensions & Tradeoffs

- **Distinct from `[[satisfaction-testing]]`, complementary to it**: satisfaction testing drives the running app through whole user journeys and asks "would a user be satisfied?"; agent-native verification instead engineers the *component* to expose its state so the check is precise and machine-readable rather than journey-level. They compose — a browser-driven journey can read published DOM contracts to verify each step — but one bets on end-to-end experience, the other on instrumented internal state.
- **It is `[[friction-as-judgment]]` answered structurally**: where that concept says fast generation removes the human review checkpoint and must be rebuilt mechanically, agent-native verification *is* one of those mechanical rebuilds — making the artifact itself legible enough for an agent (and a human dashboard) to judge, the verification-side counterpart to "agent-legible codebases."
- **State contract quality bounds the trust**: an agent verifies exactly what the component chooses to publish, so an incomplete or wrong data contract lets a real defect pass while the published state looks correct — the recurring "quality of the check bounds the trust" caveat of `[[self-verification-loop]]` and `[[holdout-validation]]`, here relocated into the artifact's instrumentation. A component that publishes a flattering but inaccurate state is the agent-native form of a verifier that verifies the wrong thing.
- **Instrumentation is standing work**: making every component publish its state and wiring the three verification runtimes is real engineering the team must build and maintain, so the long-running-agent payoff is paid for up front — the same build-cost-vs-reliability bet as embedding any verification scaffold.
- **Same-context verifier caveat persists**: an agent reading state it (or a sibling agent) produced can still share blind spots, so agent-native verification reduces ambiguity about *what happened* without supplying the model-diversity defense of `[[cross-model-critique]]` or the context-withholding of `[[holdout-validation]]`.
- **Vantage caveat**: the framework and the Anthropic-internal-practice claim come from a single first-party workshop demonstrating one stack (Storybook + Playwright MCP + React), so it is illustrative of the *pattern* — verification embedded as machine-readable artifact state — rather than an independently measured result; the durable idea is publishing internal state as an agent-readable contract, not the specific tools.
