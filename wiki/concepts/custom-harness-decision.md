# Custom Harness Build-vs-Buy

Custom harness build-vs-buy is the engineering decision of *when to wrap your own control system around an agent loop* rather than adopting an existing coding agent. The framing that makes the decision legible is that the model is the **engine** but the harness is the **vehicle**: the agent loop (prompt → think → call a tool → observe → repeat) is a small, near-universal core, and the harness is the control plane built around it that decides which model runs, which tools are exposed, how results are fed back, how context is managed, what guardrails apply, and when the loop stops. Because that core loop is shared, *most of the product differentiation between coding agents lives in the harness, not in the underlying LLM* — Claude Code, OpenCode, and Cursor are all sophisticated harnesses around the same fundamental loop. The practical rule that follows is asymmetric: reach for an off-the-shelf harness by default, and build a custom one only when specific constraints make the off-the-shelf control plane insufficient.

## Key Mechanics

- **Harness as control plane, model as engine**: the harness governs model selection, the available tool surface, how tool results are returned to the model, context management to avoid blowing the token window, guardrails and safety, and the loop's termination — the wrapper that turns a raw `[[agentic-loop]]` into a usable product. The differentiation among coding agents is in this layer, which is why nominally similar tools feel very different to use (the workflow-fit point of `[[agentic-coding-tool-selection]]`).
- **Default to off-the-shelf**: if the goal is to get work done and an existing agent meets the need, using a ready-made harness is the right call — building custom is over-engineering when the standard control plane already fits.
- **Build custom only for specific constraints**: a custom harness is justified by calling private internal APIs, enforcing a specific tool-execution order, satisfying compliance or audit requirements (scoped permissions, mandatory logging), embedding the agent directly into your own product, or needing strong control over policy, safety, and observability that an off-the-shelf agent does not expose.
- **The custom core is compact**: the loop itself — set up a system prompt plus the user task, enter a `while` loop, call the LLM, execute any tool calls, append results to history, repeat until a stop condition — is surprisingly small and functional; the engineering effort is in the control plane around it, not the loop.
- **Robustness practices for a custom build**: design narrow tools with single, clear jobs and precise descriptions (`[[robust-tool-design]]`); always feed tool-call failures back to the model so it can self-correct; track budgets (iterations, tokens, cost) from day one to prevent runaway spend; manage context with strategies like history summarization/compaction (`[[context-compaction]]`); and log everything (model responses, tool calls, results) for debugging and audit.
- **Minimal-core, extensible design**: a custom harness can stay deliberately small at the center and push advanced behavior (plan mode, permission gates) into optional extensions — the corpus's Go example, Kit (inspired by Pi), keeps a radically small core of eight tools (bash, read, write, edit, grep, find, ls, spawn sub-agent), stays provider-agnostic across Anthropic/OpenAI/Gemini/Ollama, ships as a single binary, and exposes itself as a CLI, an embeddable SDK, and an LSP server — composability over complexity.

## How It Appears in the Corpus

The Ed Zynda ("What The Func?") tutorial "I built an Agent Harness in Go" explains the agent loop as a simple `while` loop with four parts (system prompt, message history, tool surface, stop condition), then defines the harness as the control plane wrapped around it — "the model is the engine, the harness is the vehicle" — arguing most coding-agent differentiation is in the harness rather than the LLM. It frames Claude Code, OpenCode, and Cursor as sophisticated harnesses over the same loop, advises using off-the-shelf agents unless specific constraints (private APIs, enforced tool order, compliance/audit, product embedding, strong policy/safety/observability control) demand otherwise, and walks through building a custom harness with narrow tools, error feedback, budget tracking, context management, and comprehensive logging. It closes with Kit, a Go harness inspired by Pi, as an example of a minimal-core, composable custom harness available as CLI, SDK, and LSP.

## Tensions & Tradeoffs

- **Speed vs. control**: off-the-shelf harnesses win on time-to-value and avoid over-engineering, while custom harnesses win only where their specific control needs (compliance, embedding, private integration) cannot be met otherwise — building custom for a need an existing agent already serves is wasted effort.
- **Distinct from `[[harness-engineering]]`**: that concept is the discipline of *pre-defining rules, environment, and reference material* to make any agent reliable ("fix the harness, not the prompt"); build-vs-buy is the prior, architectural decision of *whether to author the control plane at all* or inherit one. They compose — a custom harness still needs harness-engineering's rules-and-constraints inside it.
- **The minimal-core bet aligns with `[[self-modifying-agent-harness]]` and `[[thin-harness-fat-skills]]`**: Kit's small-core/extensions design and its Pi lineage are the same thin-loop philosophy, betting that a compact, observable, provider-agnostic core plus optional extensions beats a heavy fixed-feature tool — and it inherits that bet's dependency on a capable model to drive the thin loop well.
- **Provider-agnosticism is the `[[model-abstraction-layer]]` applied at the harness**: a custom harness that supports multiple LLM backends is the vendor-portability hedge built into the control plane, but it presupposes rough capability parity across providers for the task.
- **"Differentiation is in the harness" cuts against pure model-shopping**: if the control plane is where the product value lives, then choosing or building a harness — not just picking the strongest model — is the real lever, the same convergence-under-feature-parity logic as `[[agentic-coding-tool-selection]]`.
- **Vantage caveat**: the framing and the Kit/Pi examples come from a single intermediate tutorial demonstrating one Go implementation, so they illustrate the *pattern* — harness as control plane, build only for specific constraints, minimal extensible core — rather than a benchmarked result.
