# Agent Alignment Artifacts

Agent alignment artifacts are short, focused intermediate documents an agent produces *before writing code* — a **design discussion** of its understanding, a **structure outline** of its intended architecture, and a **vertically-sliced plan** with testable checkpoints — whose purpose is to make the agent's thinking legible and *correctable* by a human (or a team) at the cheapest possible point. The premise is that the costliest mistakes are misunderstandings baked silently into a long plan and then into code; surfacing the agent's mental model as a small, reviewable artifact lets the human perform "brain surgery" on the approach — fix a wrong pattern, resolve an open question, reorder the work — while it is still prose. It is the methodology that emerged when a single monolithic "plan" prompt proved both unreliable (it overran the `[[instruction-budget]]`) and opaque (the human never saw the agent's assumptions before code existed), splitting planning into staged artifacts each scoped for genuine alignment.

## Key Mechanics

- **Design discussion (~200 lines)**: the agent brain-dumps its understanding of the problem — the desired end state, the existing patterns it intends to follow (so the human can *correct a bad pattern before it propagates*), the decisions it has resolved, and the open questions it still has. This forces early human-agent alignment and is where the human edits the agent's *approach* rather than its output — the artifact form of `[[collaborative-prompt-elicitation]]`, run from the agent's side.
- **Structure outline (~2 pages)**: a high-level map of the implementation — the development phases, the order of changes, and the testing strategy — functioning like a "C header file" for the work: a declaration of architecture the human can review for sound thinking before any implementation fills it in. It is the cheap checkpoint where architectural divergence is caught.
- **Vertical plans over horizontal plans**: the plan is sliced *vertically* — a thin end-to-end path through every layer at once (e.g. mock API → frontend → service → database) with a checkpoint to test after each slice — rather than *horizontally* (all the database work, then all the service work, then all the frontend). Vertical slices integrate and are debuggable at each checkpoint, whereas horizontal plans defer all integration to the end, where failures are hardest to localize.
- **Artifacts as a team-review surface**: because the design discussion and structure outline are short and focused, they can be reviewed by a *team* early — mirroring a strong team's architecture-review process — which front-loads correction and sharply shortens the eventual code review, the same shift-review-upstream logic as `[[spec-review-loop]]` and `[[multi-agent-code-review]]` applied to the design rather than the diff.
- **Read and own the resulting code**: the methodology explicitly reverses an earlier "don't read the code" stance — for production-grade, regulated, or user-facing software the human must read and *own* the generated code, because not doing so accumulated "slop" that required rework and wholesale replacement. The target is a *2–3x speedup at human-level quality*, not a 10x flood of unowned output.

## How It Appears in the Corpus

The Dexter Horthy (MLOps.community) talk recounts the evolution of the Research-Plan-Implement (RPI) coding-agent methodology into a successor named "Crispy" (Questions, Research, Design, Structure, Outline, Plan, Work, Implement, PR). RPI's original advice — don't read the code, rely on a single long plan file — produced ~50% rework ("slop") and worked only for low-complexity greenfield tasks. The talk attributes the failure partly to the `[[instruction-budget]]` and partly to a lack of early human alignment, and introduces the design discussion (a ~200-line brain-dump of understanding, patterns, decisions, and open questions for human correction), the structure outline (a ~2-page "C header file" of phases, change order, and testing), and vertical plans with checkpoints (mock API → frontend → service → database) over harder-to-debug horizontal plans. It frames these short, focused artifacts as enabling early team review and "brain surgery" on the agent's approach before code, and explicitly recants the "don't read the code" position in favor of reading and owning code for reliable 2–3x gains at professional quality.

## Tensions & Tradeoffs

- **Lighter than `[[spec-driven-development]]`, heavier than `[[plan-first-workflow]]`**: a plan-first loop has one plan-review checkpoint; spec-driven development imposes ordered hard gates; alignment artifacts sit between — multiple *staged* lightweight documents (understanding, then architecture, then sliced plan), each a correction point, but without the rigid no-shortcuts gating. The bet is that several cheap alignment passes beat either one coarse checkpoint or a heavyweight gate sequence.
- **Vertical-slice debuggability vs. layer efficiency**: vertical slices integrate continuously and localize failures at each checkpoint, but they require scaffolding (mocks, stubs) to exercise an end-to-end path before all layers exist; a horizontal plan avoids that scaffolding cost at the price of deferring every integration risk to the end. The methodology bets debuggability is worth the mock overhead.
- **"Read and own the code" reverses the outsourcing instinct**: owning the generated code is exactly the `[[friction-as-judgment]]` discipline — re-inserting human review where fast generation removed it — and the guardrail against `[[vibe-coding-vs-agentic-engineering|vibe-coded]]` slop, but it caps the speedup at 2–3x rather than the headline 10x, an explicit trade of throughput for quality and ownership. It is the generated-code counterpart to `[[code-comprehension-over-generation]]`'s comprehend-first stance.
- **Artifact quality bounds the alignment**: an agent's brain-dump only helps if it surfaces the *real* assumptions and the human actually catches the wrong ones — a confidently-stated but subtly wrong design discussion can ratify a flawed approach, the same "quality of the artifact bounds the trust" caveat and a place where human `[[engineering-taste]]` must still bite.
- **Distinct from `[[exploratory-spec-discovery]]`**: that pattern *discovers* a spec by letting the agent explore a throwaway branch until a verifiable goal is hit; alignment artifacts instead *author* the design up front for human correction. One is spec-by-exploration, the other spec-by-early-alignment — complementary for, respectively, unknown-shape and known-shape problems.
- **Vantage caveat**: the artifact names, line/page sizes, and the 2–3x figure come from one practitioner's iterated methodology, so they are illustrative of the *pattern* — staged, correctable, human-legible pre-code artifacts plus vertical slicing — rather than measured results.
