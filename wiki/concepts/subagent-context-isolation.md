# Subagent Context Isolation

Subagent context isolation is the pattern of breaking a complex problem into smaller, focused jobs and handing each to an *independent* agent session, which performs the specialized work in its own context and returns only a concise summary to the main session. Its defining benefit is not parallelism but *context hygiene*: the messy intermediate work — file reads, tool output, dead ends — stays inside the subagent's window, and only the distilled result crosses back, so the primary conversation stays clean and its finite context window stays lean. The subagent acts as a context firewall around a sub-task.

## Key Mechanics

- **Decompose into focused jobs**: a large task is split into smaller, well-scoped sub-tasks, each given to a dedicated subagent rather than worked inline in the main thread.
- **Independent session, isolated context**: each subagent runs as its own session with its own context window, so the tokens it spends exploring and iterating do not accumulate in the parent's window.
- **Summary-return, not transcript-return**: the subagent reports back a concise summary of its result, not its full working transcript — the mechanism that keeps the main conversation high-signal and defers the `[[context-rot]]`/`[[context-decay]]` that a single ever-growing thread would suffer.
- **Split-and-merge fan-out**: within a single session the main agent can split a task into independent pieces and fan the work out to multiple subagents at once (the corpus cites up to ~10 simultaneously, customizable), each working its slice in its own context before reporting back so the main agent merges the results — the in-session parallelization variant of the pattern (e.g. researching several competitors concurrently).
- **No direct subagent-to-subagent communication**: a defining limit of split-and-merge is that subagents cannot talk to each other — *all* information must funnel through the main agent, which is the sole integration point. This funnel is the bottleneck that `[[agent-team-collaboration]]` removes by letting agents communicate directly, trading context hygiene for richer coordination.
- **Built-in specialized subagents**: the agent can route work automatically to purpose-scoped subagents — e.g. a fast, cheap, read-only *explore* agent for file scanning, a *plan* agent for codebase research, and a read/write *general-purpose* agent for heavy multi-step work — each in its own context window, an automatic instance of matching capability and cost to task difficulty (`[[per-node-model-routing]]`).
- **Spawning is just another tool call**: dispatching a subagent is itself a tool invocation (an `agent tool`) within the agent's normal tool registry, so delegation is architecturally consistent with every other action in the `[[agentic-loop]]` rather than a special control path — the orchestrator–worker dispatch of `[[distributed-systems-literacy]]` expressed through the same interface as any file read or shell command.
- **Parallel orchestration of specialized workers**: the main agent can spawn and orchestrate multiple subagents, each running its own focused loop on a distinct aspect of the task (e.g. test generation, security scanning) and reporting findings back to the orchestrator — the worker-node pattern of a distributed system applied to agent work, and the substrate for a builder-validator chain (one subagent builds, another checks) that bakes quality control into the fan-out.
- **Complements orchestration**: subagents are a natural unit for a `[[deterministic-workflow-orchestration]]` node or for `[[multi-agent-code-review]]`, where specialized reviewers each return findings rather than flooding one context.

## How It Appears in the Corpus

The ByteByteAI "12 Claude Code Features" overview presents sub-agents as a way to tackle complex tasks: Claude Code splits a problem into smaller, focused jobs, each subagent operating as an independent session that performs specialized work and returns a concise summary to the main session — explicitly framed as keeping the primary conversation clean and the context window lean.

The ByteMonk "Claude's Internal Architecture Revealed" analysis corroborates the pattern from the architecture side: for tasks too complex for a single thread of work, the main agent dynamically spawns and orchestrates sub-agents, each running its own focused loop on a specific aspect (e.g. test generation, security scanning) and reporting findings back to the main orchestrator. It stresses that spawning a sub-agent is *merely another tool call* (`agent tool`) within the system's registry — preserving architectural consistency — and frames the whole arrangement as mimicking a distributed system in which an orchestrator delegates to specialized workers (`[[distributed-systems-literacy]]`).

The Simon Scrapes "Every Claude Code Workflow Explained" tutorial frames the pattern two ways. First it describes the *built-in* specialized subagents Claude routes to automatically even in a basic session — a fast, cheap, read-only "explore" agent (on a Haiku-class model) for file scanning, a "plan" agent for codebase research, and a read/write "general purpose" agent (on a Sonnet-class model) for heavy multi-step work — each with its own context window so the main conversation does not bloat. It then names "split and merge" as the explicit fan-out pattern: within one session the main agent splits a task into independent pieces, dispatches them to up to ~10 subagents that each work in isolation, and merges their reports — emphasizing the key limitation that the subagents cannot communicate directly and all information funnels through the main agent, and that this enables builder-validator chains for built-in quality checks.

## Tensions & Tradeoffs

- **Distinct from `[[parallel-isolated-agents]]`**: that concept runs many sessions *concurrently* in isolated workspaces under one conductor for throughput; subagent context isolation is about *context-window economy* via summary-return delegation, and a subagent need not run in parallel at all. The two compose — parallel subagents both isolate context and add concurrency.
- **The funnel is a feature and a ceiling**: routing all subagent output through the main agent is exactly what keeps the parent the clean integration point, but it caps coordination — work that needs cross-talk between workers cannot get it, which is the precise limitation `[[agent-team-collaboration]]` lifts (at a 4–7x token cost) by allowing direct peer communication.
- **The summary is a lossy interface**: returning only a concise summary is exactly what keeps the parent lean, but it can drop detail the main agent later needs — the same coverage gap as summarized vs. verbatim storage in `[[progressive-disclosure-retrieval]]`, here at the delegation boundary.
- **Decomposition quality bounds the result**: isolation only helps if the sub-task was scoped well; a poorly carved subagent job returns a summary that answers the wrong question, and the parent cannot see the working that would reveal the error.
- **Coordination overhead**: spawning and reconciling many subagents is itself work, and a context-firewall that hides too much can make the main agent's view of the task incomplete.
