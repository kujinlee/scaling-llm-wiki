# Ralph Loop

The Ralph loop is the pattern of wrapping an autonomous, headless agent run in an *outer* loop that re-issues the **same prompt** over and over while the agent builds files, so the agent keeps iterating on a task across many fresh runs until it converges on completion. It is distinct from the inner `[[agentic-loop]]` that the model drives *within* a single session: the Ralph loop is an operator-imposed repetition layered *around* that session, relaunching the agent with an unchanged instruction so each pass picks up where the last left off — not by remembering the conversation, but by reading the work already written to disk. Its defining benefit is *self-correction through repetition*: each re-run prompts the agent to review its own prior work, notice what it omitted or got wrong, and refine it — so a deliberately "dumb" loop of the simplest possible kind produces robust output that a single one-shot pass would not.

## Key Mechanics

- **Outer loop over a fixed prompt**: the operator re-feeds one unchanging prompt each iteration; the agent does not need a new instruction because the remaining work is implied by what is and isn't yet on disk. Each run advances the build a little further, and the implementation can be as crude as a `while true` shell script around a headless agent invocation.
- **Self-correction is the core payoff**: repeatedly asking the agent the *same* thing makes each pass a review of the last — the agent catches its own omissions and errors and iteratively sharpens the output. This was the original motivation (named after Ralph Wiggum, for its deliberate dumbness) and remains valuable even as newer models self-correct better within a single run.
- **Files are the state between runs**: because every iteration starts a fresh agent context, the *filesystem is the memory* — the agent reads the existing files, sees what is incomplete, and extends them, so durable progress survives the loss of conversational context that would otherwise end with each run.
- **Backlog-driven prioritization scales the loop**: the loop's real power emerges over a *list* of tasks. Rather than a brittle, waterfall-style pre-orchestration of many parallel agents, the operator instructs the agent each pass to "pick the next most important ticket" from a backlog and work it — letting a modern LLM understand dependencies and dynamically prioritize, so a single simple loop autonomously grinds through an entire project's work. Simplicity beats elaborate orchestration here.
- **Feedback defines "good"**: for open-ended tasks (e.g. newsletter generation) the loop needs an explicit notion of quality to climb toward — supplied by techniques like audience simulation and iterative AI self-critique, so the loop has a target rather than just re-running blindly.
- **Built on headless autonomous execution**: it sits on top of a CLI / headless agentic mode (the corpus's `-P` flag, or a productized `loop` command) that runs the full decide-execute-feedback cycle with no per-step human approval — the unattended substrate the outer loop repeats.
- **Runs until completion (or an operator stop)**: the loop continues feeding the prompt while files are being built, ideally terminating when the task is realized — making it a way to grind a large, multi-step build to completion without a human re-prompting after each pass.

## How It Appears in the Corpus

The Simon Scrapes "Every Claude Code Concept Explained for Normal People" beginner overview presents CLI mode (the `-P` flag) as a way to run a full agentic loop independently of human interaction or approval prompts, and frames the "Ralph Loop" as extending that: continuously feeding the same prompt as files are built, allowing the agent to iterate on the task until completion.

The Chris Parsons (Cherrypick, "AI Engineer" channel) workshop "Ralph Loops: Build Dumb AI Loops That Ship" is the authoritative treatment. It frames a paradigm shift away from brittle, complex N8N-style orchestration toward simple, robust, self-correcting AI loops — arguing that because modern LLMs already operate on an internal loop, a single adaptive agent in a dumb outer loop beats a hand-wired multi-agent workflow. It demystifies the "Ralph loop" (named after Ralph Wiggum) as repeatedly asking the AI to do the same task so it reviews and refines its own prior work, walks through a hands-on `while true` script around Claude Code on a Pomodoro-timer project (single feature ticket first, then a `loop` command for continuous execution), and unlocks scale by having the AI "pick the next most important ticket" from a backlog rather than orchestrating parallel agents in a waterfall. It then covers the advanced concerns real loops demand: defining "good" output through audience simulation and AI self-critique, sandboxing the agent against security risks (`[[agent-sandboxing]]`), knowledge management via personal "vaults," and the human's residual role as strategist and bottleneck-resolver rather than task-executor.

## Tensions & Tradeoffs

- **No built-in verifier means it can spin without progress**: re-feeding a prompt does not supply a stopping rule or a correctness signal, so a Ralph loop inherits the unbounded-by-construction problem of the bare `[[agentic-loop]]` — it can iterate indefinitely or thrash without converging. The discipline that fixes this is a *verifiable goal* (`[[hill-climbing]]`, `[[exploratory-spec-discovery]]`), an explicit definition of "good" output (`[[custom-eval-systems]]`, audience simulation), and an embedded `[[self-verification-loop]]`, so each pass measurably advances toward a checkable finish line rather than just running again.
- **Distinct from a `[[generator-verifier-loop]]`**: that pattern adds a separate supervisor agent that asks "is the goal met?" after each step; the raw Ralph loop has *no checker* — it simply repeats the same prompt, trusting the filesystem state and the agent's own judgment to drive completion, which is why it is the cruder, more runaway-prone cousin.
- **Simplicity over orchestration is a deliberate bet**: the workshop's claim that a dumb single-agent loop beats elaborate parallel orchestration runs against the heavier `[[deterministic-workflow-orchestration]]` direction — both ship work autonomously, but Ralph bets that LLM adaptivity (e.g. dynamic backlog prioritization) makes most pre-wired coordination unnecessary, while orchestration bets that critical steps must be enforced rather than left to the model. The presenter's own arc — from brittle N8N workflows to simple loops — is the argument for the simpler pole.
- **Filesystem-as-state accumulates cruft**: because each iteration builds on whatever the prior runs left behind, the agent can compound its own bad patterns across passes — the entropy that `[[ai-garbage-collection]]` exists to prune, here amplified by repetition.
- **Autonomy, cost, and security gated by trust**: an unattended loop re-running a headless agent with no approval is exactly where `[[graduated-autonomy]]` applies — it consumes tokens continuously, acts without a human backstop, and (with internet access and credentials in play) widens the security blast radius that `[[agent-sandboxing]]` exists to contain. It suits low-stakes, recoverable, well-scoped builds before higher-stakes ones, and is a building block beneath always-on `[[agent-as-infrastructure]]` and the autonomous loops of a `[[dark-factory]]` rather than a safe default.
- **Vantage caveat**: the durable idea is the *same-prompt outer loop with files as state plus self-correction through repetition*, not a specific command or tool; the beginner source gives it a one-line treatment and the workshop demonstrates it on small projects, so the pattern is illustrated more than independently benchmarked at scale.
