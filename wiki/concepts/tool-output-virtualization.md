# Tool-Output Virtualization

Tool-output virtualization is the technique of inserting a layer between an agent and its tools that *intercepts* large tool outputs, indexes them into an external store, and returns to the agent only a tiny reference (a confirmation that the data was captured and is queryable) instead of the raw output itself. It attacks a specific, mechanical cause of context exhaustion: every tool call normally dumps its full result straight into the finite context window — a single browser snapshot or a batch of issues can consume a large fraction of the window before the agent writes a line of code — so the agent runs out of room, pays for the tokens, and forgets earlier work. By keeping the bulk data outside the window and letting the agent retrieve only what it needs through queries, virtualization frees the context window for reasoning rather than for verbatim tool dumps. It is the plumbing-level counterpart to `[[subagent-context-isolation]]`: where a subagent keeps messy intermediate work in a *separate session*, virtualization keeps bulky tool output in a *separate store* the same session can query.

## Key Mechanics

- **Intercept before the window**: the layer sits between the agent and the operating system / tools, capturing each tool's output *before* it is injected into the context window — so the raw blob never costs context tokens in the first place.
- **Index into an external store**: captured output is written to a local indexed database (the corpus's instance is SQLite with FTS5 full-text search), turning a one-shot dump into a persistent, searchable resource — the same store-then-retrieve shape as a `[[filesystem-memory-store]]`, applied to transient tool output rather than long-term memory.
- **Return a reference, not the payload**: the agent receives only a small handle (e.g. confirmation that a 5,000-line log file is indexed), with the corpus reporting reductions like a 56 KB browser snapshot to ~299 bytes (~99%) — the agent then *queries* the store for the specific patterns it needs (e.g. error codes and the IP addresses behind them) rather than parsing the whole file.
- **Query-on-demand is agentic retrieval**: because the agent decides what to pull from the indexed store and how often, virtualization is `[[agentic-search]]` pointed at the agent's *own* tool outputs — the data is fetched into the window only when a query demands it, the loading-side discipline shared with `[[progressive-disclosure-retrieval]]`.
- **Hook-driven session continuity**: lifecycle `[[lifecycle-hooks]]` monitor file edits, git operations, and sub-agent tasks, and when the conversation `[[context-compaction|compacts]]` the layer builds a small priority-tiered snapshot and re-injects it — a "save checkpoint" that also records past errors and decisions so the agent does not repeat failed attempts after its context resets.
- **Distributed as an MCP server**: the capability ships as a `[[model-context-protocol]]` server / plugin (installable via marketplace or npm) usable across multiple agents (Claude Code, Gemini CLI, VS Code Copilot) — the virtualization layer as a portable, tool-agnostic add-on.

## How It Appears in the Corpus

The Better Stack "Claude Code is Expensive" tutorial introduces "Context Mode" as a virtualization layer that fixes context-window exhaustion. It diagnoses the problem mechanically — each tool call dumps its full output into the window (a Playwright snapshot at 56 KB, twenty GitHub issues at 59 KB), consuming up to ~70% of a 200K window before any code is written and causing the agent to forget tasks and decisions within ~30 minutes. Context Mode intercepts and indexes those outputs into a local SQLite FTS5 database, returning only a reference (56 KB → 299 bytes; a large analytics CSV → 222 bytes). Its demonstration indexes a 5,000-line `access.log` and lets Claude query the database for 500-error patterns and associated IPs without parsing the raw file, with a `stats` check showing token savings (25% on a small demo, framed as scaling to ~100K tokens saved on large projects). Beyond token savings, the video stresses *session continuity*: hooks track every file edit, git operation, and sub-agent task, and on compaction a priority-tiered snapshot (under ~2 KB) is rebuilt and re-injected as a checkpoint — hypothetically extending an effective session from ~30 minutes to ~3 hours and preventing the agent from repeating past errors after a reset.

## Tensions & Tradeoffs

- **Retrieval quality becomes the bottleneck**: virtualization only helps if the agent can find the relevant data in the indexed store — an FTS5 full-text index matches literal terms, not meaning, so it inherits the "literal search vs. semantic retrieval" limit of `[[filesystem-memory-store]]` and the "retrieval quality is the new bottleneck" caveat of `[[agentic-search]]` and `[[persistent-agent-memory]]`. A blob that is captured but never surfaced by a query is effectively lost.
- **It addresses `[[context-rot]]` at the plumbing layer**: where `[[context-rot]]` is dilution by standing material and `[[context-compaction]]` condenses live history, virtualization prevents *transient tool output* from entering the window at all — a complementary third lever on the same finite budget, attacked at the tool-call boundary rather than the instruction file or the conversation history.
- **A reference is a lossy interface**: returning a handle instead of the payload keeps the window lean but means the agent acts on data it has not fully read — the same summary-vs-transcript tradeoff as `[[subagent-context-isolation]]`, so an agent that queries the wrong slice gets a confident answer grounded in data it never inspected.
- **Another operational surface and trust layer**: the virtualization layer is an external store plus hooks plus an MCP server running with the agent's authority — infrastructure to install, secure, and trust, re-raising the bundled-integration trust surface of `[[model-context-protocol]]` rather than a free optimization.
- **Vantage and figures are illustrative**: the token-saving percentages and the 30-minute-to-3-hour session-extension claim come from a vendor/tutorial demo ("hypothetically"), not independent measurement, so the durable idea is the *intercept-index-reference* pattern, not the specific numbers.
