# Embedded Guardrails vs. Triggered Skills

Embedded guardrails vs. triggered skills is the architectural distinction between two ways of shaping an agent's behavior: a *constant, always-loaded constraint* baked into the context file so it applies to every action (an embedded guardrail, functioning like the model's inherent "personality"), versus a *task-specific procedure* invoked only when a particular kind of work is at hand (a triggered skill). The defining contrast is *when the behavior is in force*: an embedded guardrail is on all the time and shapes how the agent does everything, while a triggered skill is dormant until a matching task explicitly activates it. The corpus's pointed conclusion is that these are complementary layers, not rivals — the optimal setup *combines* them, embedding behavioral guardrails as a constant foundation and triggering structured skills per task.

## Key Mechanics

- **Embedded guardrail = constant constraint as personality**: principles loaded directly into `[[claude-md]]` apply to every session and every action, behaving like a built-in personality the model cannot step outside of. The corpus's instance is `[[surgical-change-discipline|Karpathy's four principles]]` (think before coding, simplicity, surgical changes, goal-driven execution) embedded into `claude.md` so they constrain all behavior rather than being summoned for a specific job.
- **Triggered skill = on-demand procedure**: a `[[procedural-knowledge-skills|skill]]` (brainstorming, planning, debugging, simplification, TDD) is invoked explicitly for a specific task and otherwise stays out of the context window — the trigger-based loading of `[[lazy-context-loading]]`, applied to whole workflows. Frameworks like G-stack, Superpower, and GSD are characterized as collections of such triggered skills (write a spec, build a to-do list, then act).
- **Different jobs, different layers**: embedded guardrails govern *how much* and *where* (restraint, scope, simplicity) — constraints that should never be off; triggered skills govern *what structured process* to run for a given task (the ordered phases of `[[spec-driven-development]]`, the red-green loop of `[[agent-tdd]]`). The corpus notes that simplicity and surgical-change guardrails are precisely the constant constraints that triggered planning frameworks do *not* supply, while planning-style guardrails overlap with those frameworks' phases.
- **The hybrid is the recommended posture**: embed the behavioral principles as a constant foundation, then direct the agent to call the right triggered skill for each task (brainstorming for a new feature, debugging for an error, planning for a multi-step job, simplification before committing, TDD for implementation). Constant behavioral constraints plus structured task-specific execution is offered as the highest-accuracy workflow.
- **Installed as bundled artifacts**: embedded guardrails ship as a single-file `claude.md` plugin (installable globally via a marketplace or per-project via a `curl` command), and merging them into an existing context file is itself an agent task — using Claude Code to resolve conflicts (duplicate H1s, irrelevant meta-framing) and keep the file concise.

## How It Appears in the Corpus

The Eric Tech video "Karpathy's Skill Just Fixed Claude Code's Biggest Problem" draws the distinction explicitly while presenting Karpathy's four coding principles. It contrasts *embedded guardrails* — the principles written into `claude.md` so they act as constant, inherent constraints like a model's personality — with *triggered skills* from frameworks such as G-stack, Superpower, and GSD, which must be explicitly invoked for specific tasks (writing a spec, creating a to-do list, then acting). It argues that the Simplicity and Surgical-Changes principles are unique always-on guardrails the triggered-skill frameworks do not cover (they teach careful work but not *how much* to do and *where*), while Think-Before-Coding and Goal-Driven principles overlap with those frameworks' planning phases. The video's recommended best practice is a hybrid: keep Karpathy's principles as embedded foundational guardrails and strategically trigger framework skills (Superpower brainstorming for new features, system debugging for errors, GSD planning for multi-step tasks, simplification before committing, TDD for features and bug fixes) as each task demands — so the agent operates inside consistent behavioral constraints while leveraging structured, task-specific execution.

## Tensions & Tradeoffs

- **Always-on costs context budget; triggered does not**: every embedded guardrail occupies the always-loaded window on every prompt, re-raising the `[[context-rot]]` dilution risk that bounds `[[claude-md]]`, whereas a triggered skill spends context only when active. So the design judgment is which behaviors are important enough to pay for constantly — a guardrail that should bite everywhere (restraint, simplicity) earns its standing cost, while a procedure needed only sometimes belongs behind a trigger.
- **A different axis than `[[thin-harness-fat-skills]]`**: that principle argues *where the intelligence lives* (a thin loop plus thick prose skills); this distinction argues *when a behavior is in force* (constant vs. on-demand). They compose — fat skills can be triggered while a thin set of guardrails stays embedded — but they answer different questions.
- **Adjacent to `[[explicit-gears]]` and `[[assistant-personalization-layers]]`**: explicit gears switch the agent's *cognitive mode* per task (a form of triggered behavior), and the personalization layers separate context/method/profile; embedded-vs-triggered is the narrower cut of *constant constraint vs. invoked procedure* that runs beneath both — guardrails are the always-on profile, gears and skills are the triggered method.
- **Guardrails are prose, not enforcement**: an embedded guardrail in `claude.md` is a constant *request*, not a mechanical block, so it shapes behavior without guaranteeing it — the same weakness as any `[[claude-md]]` instruction relative to a deterministic gate, and the reason the hybrid still leans on triggered verification skills (TDD, review) rather than trusting the personality alone.
- **Vantage caveat**: the embedded/triggered framing and the hybrid recommendation come from a single intermediate tutorial built around one set of principles, so it illustrates the *pattern* — constant guardrails plus on-demand skills — rather than a measured result; the durable idea is matching a behavior's loading tier to whether it should be always-on or task-triggered, not the specific principles or frameworks named.
