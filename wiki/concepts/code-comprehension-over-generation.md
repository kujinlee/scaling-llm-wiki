# Code Comprehension Over Generation

Code comprehension over generation is the claim — grounded in a working engineer's audit of her own AI usage — that the most impactful role for AI in a large, complex, fast-moving codebase is not *writing* code but *understanding* it. The common narrative casts AI as a code generator, yet in practice the binding constraint for an engineer on a 15-year-old system is keeping an accurate mental model of code that changes faster than any human can track. AI excels precisely there: it acts as an "always-on senior engineer" that never tires of questions, collapsing what used to be tedious manual investigation or a cross-timezone wait for a colleague into an immediate, structured explanation. The defining move is to deliberately point AI at comprehension first and let generation follow only once understanding is in place — "comprehend first, code later."

## Key Mechanics

- **The usage audit reveals the real split**: a self-analysis (`[[usage-pattern-analysis]]`) of one engineer's prompts found ~67% were for *comprehension* and only ~2% for code *generation* — a concrete, surprising inversion of the assumption that AI's value to developers is writing code. The metric reframes where the leverage actually is.
- **The codebase as a moving target**: comprehension is hard because the system never holds still — the corpus cites ~100 PRs merged daily, with new components, deprecations, and lint rules constantly arriving, so an engineer's mental model decays continuously and must be re-established on demand rather than learned once.
- **Structured multi-mode comprehension skill**: rather than ad-hoc questions, the comprehension need is systematized into a custom `[[procedural-knowledge-skills|skill]]` ("catch me up") — a stored, structured prompt that splits comprehension into explicit exploration modes: **Architecture**, **Convention**, **Feature Trace**, **Syntax**, **Testing**, and **History**. Each mode returns structured output (organograms, tables) sized to the question, so onboarding to a new repo or gaining enough context to review a colleague's PR becomes a repeatable, fast operation.
- **AI as always-on senior engineer**: the agent answers root-cause and historical-decision questions instantly and without fatigue — replacing the bottleneck of waiting for the one person who remembers why a thing was built — so the engineer's mental model stays aligned with the code.
- **Comprehension is the precondition for safe generation**: the practice exists inside a larger AI-native workflow (a code-review agent on PRs, a bot that turns bug reports into fix PRs — see `[[agentic-issue-resolution]]` and `[[multi-agent-code-review]]`); comprehension is what lets the human steer those generators correctly rather than rubber-stamp them.

## How It Appears in the Corpus

The Priscila Andre de Oliveira (Sentry, "AI Engineer" channel) talk "Comprehend First, Code Later" recounts a Senior Software Engineer's shift since late 2023 from writing code to primarily *prompting*, managing a team of AI agents inside Sentry's 15-year-old observability codebase relied on by ~100,000 organizations. She reports that an audit of her own AI usage showed ~67% of prompts were for comprehension and only ~2% for generation, countering the perception that AI mainly writes code. She presents a personal "catch me up" skill that structures comprehension into six modes (Architecture, Convention, Feature Trace, Syntax, Testing, History) returning organograms and tables, and situates it among Sentry's internal AI tools (Abacus usage tracking, Warden PR review, Junior bug-to-PR bot). Citing Armin Ronacher's and others' concerns, she argues the biggest unlock from AI in a large codebase is enhanced comprehension, which lets developers hold a real mental model and ship "keynote code" rather than "slop code."

## Tensions & Tradeoffs

- **It is the daily-practice face of `[[thinking-vs-understanding]]`**: Karpathy's distinction holds that reasoning can be outsourced but comprehension cannot; comprehension-first AI use is the concrete workflow that honors it — the human deliberately spends AI on *settling the code into their own head* rather than letting the agent both write and "understand" on their behalf.
- **The guardrail against `[[vibe-coding-vs-agentic-engineering|vibe coding]]`**: shipping AI-generated code without comprehending it is exactly the "slop code" failure mode; routing AI toward comprehension is how an engineer keeps the quality bar ("keynote code") while still moving fast — the same speed-vs-responsibility bet, resolved by re-inserting human understanding where `[[friction-as-judgment]]` says fast generation removed it.
- **Comprehension quality bounds the trust**: an AI explanation of unfamiliar code can be confidently wrong, so a structured comprehension skill *accelerates* mental-model building but does not certify it — the engineer must still verify the explanation against the real system, the recurring "quality of the artifact bounds the trust" caveat, here aimed at the explanation rather than a verifier.
- **Distinct from `[[codebase-knowledge-graph]]`**: that concept builds a persistent, auto-generated structured representation of the code; comprehension-first AI use is the *human practice* of querying for understanding on demand (often via a prompt skill), which can run over a knowledge graph or directly over the repo. One is the artifact, the other is the working habit.
- **The skill is a stored procedure, not a permanent map**: "catch me up" is a `[[procedural-knowledge-skills|skill]]` whose modes encode *how to ask*, but on a target that changes ~100 PRs a day each answer is a fresh snapshot — comprehension must be re-run as the code moves, so the practice trades a one-time onboarding for a standing, repeated cost.
- **Vantage caveat**: the 67%/2% split is one engineer's self-audit at one company, not a controlled measurement — the durable idea is the *comprehension-over-generation inversion* and the structured-comprehension-skill pattern, not the specific percentages.
