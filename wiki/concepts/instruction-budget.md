# Instruction Budget

The instruction budget is the principle that a frontier LLM can *consistently* obey only a bounded number of instructions in a single call — the corpus cites roughly **150–200** — and that crossing this ceiling does not produce a graceful degradation but inconsistent, unpredictable adherence: the model silently skips steps, ignores constraints, and drops behaviors the prompt explicitly demanded. Crucially the budget is *shared* across everything loaded into the call — the user prompt, the system prompt, and the tool definitions all spend from the same allowance — so a single planning prompt carrying 85+ instructions, stacked on top of system and tool instructions, blows past the limit and yields plans that omit critical steps (presenting design options, soliciting feedback). The remedy is not a better-worded prompt but *fewer instructions per call*: decompose a monolithic prompt into a sequence of small, focused prompts, each kept well under the budget (the corpus targets **under ~40 instructions**), so each call has room to obey everything it is given.

## Key Mechanics

- **A hard ceiling on reliable adherence**: instruction-following is consistent only up to ~150–200 instructions; beyond that the model obeys an unpredictable subset, so a prompt that *contains* a step is not a guarantee the step runs. This is why "the model skipped the part where it asks for feedback" is a budget symptom, not a wording bug.
- **The budget is shared and cumulative**: the user prompt is not the only consumer — the system prompt and every tool definition also occupy the budget, so a prompt that looks modest in isolation can still overflow once the always-present machinery is counted. Token-cheap is not instruction-cheap.
- **Decompose, don't reword**: the fix for an overstuffed prompt is structural — split one 85-instruction planning step into several sub-prompts (e.g. a design-discussion prompt, a structure-outline prompt, a plan prompt), each under ~40 instructions, so each call can fully comply. This is the prompt-level analogue of why a `[[harness-engineering]]` harness chains many well-scoped calls rather than carrying one ballooning instruction set.
- **"Control flow for control flow"**: rather than asking one overburdened prompt to branch internally on every case, use real program control flow (classify the input with an `if`/router, then dispatch it to a specialized, narrowly-scoped prompt) — moving branching logic out of the instruction set and into deterministic code, which frees budget for the actual task. This is the instruction-budget rationale beneath `[[deterministic-workflow-orchestration]]`'s node-per-step decomposition and `[[per-node-model-routing]]`'s classify-then-act split.
- **The "magic words" symptom**: when users find they must include specific incantations ("work back and forth with me") to get a behavior the prompt should already produce, that is evidence the budget is exhausted — the behavior was specified but crowded out, and the user is paying instruction tokens to re-prioritize it. It is a design flaw in the prompt's instruction load, not user error.
- **The "dumb zone" of the context window**: a related but distinct degradation — results decline once context-window usage passes roughly **40%**, so even a well-budgeted set of instructions suffers if it rides inside a bloated window. Smaller, focused contexts keep both the instruction count and the window occupancy low, the same lean-window discipline as `[[context-rot]]`.

## How It Appears in the Corpus

The Dexter Horthy (MLOps.community) talk on the evolution of the Research-Plan-Implement (RPI) coding-agent methodology diagnoses the instruction budget as the root cause of RPI's unreliable planning. It reports that frontier models follow only ~150–200 instructions consistently, that RPI's single monolithic planning prompt carried 85+ instructions (before counting system prompts and tools), and that this overload made the model skip steps like presenting design options or gathering feedback. The fix that defines the successor "Crispy" workflow is to break the monolithic planning step into separate sub-prompts — Design Discussion, Structure Outline, Plan — each under ~40 instructions, dramatically improving adherence, and to use "control flow for control flow" (classify input, then route it to a focused prompt) instead of one overburdened prompt. It also revives the "dumb zone" caution that context windows degrade beyond ~40% usage, reinforcing smaller, focused contexts.

## Tensions & Tradeoffs

- **Decomposition cost vs. adherence**: splitting one prompt into several is more orchestration to author and maintain (routing logic, handoffs between sub-prompts), the same authoring overhead as `[[deterministic-workflow-orchestration]]` — but it is the price of reliable instruction-following, since a single big prompt's apparent simplicity buys silent non-compliance.
- **Distinct from `[[context-rot]]` and `[[context-decay]]`**: those name *window-occupancy* failure modes (dilution by volume, loss over length); the instruction budget is a *count-of-directives* limit that bites even in a near-empty window. The "dumb zone" 40% rule connects the two — a budgeted prompt still degrades inside a bloated context — but the budget itself is about how many things you ask, not how many tokens you load.
- **A budget for prompts is a budget for tools too**: because tool definitions spend from the same allowance, the instruction budget reinforces `[[tool-integration-hierarchy]]`'s and `[[robust-tool-design]]`'s warning that heavy, verbose tool schemas are not free — every tool added quietly narrows the room left for task instructions.
- **The numbers are rules of thumb, not measured constants**: 150–200 instructions, the ~40 sub-prompt target, and the 40% dumb-zone threshold are practitioner heuristics from one talk, varying by model, task, and how the instructions are written — the durable idea is *bounded, shared, count-based instruction-following capacity*, not the specific figures.
- **It reframes prompt failure as a structural problem**: rather than chasing a "magic prompt" (the wasted-effort trap `[[prompt-engineering]]` also warns against), the instruction budget says an unreliable prompt is usually *overloaded*, redirecting the fix from better wording to fewer directives — a `[[theory-of-mind-for-llms]]` insight about how the model actually allocates attention across its instructions.
