# LLM Knowledge Wiki

An LLM knowledge wiki is a structured, retrieval-backed knowledge base built specifically so an agent can pull the *relevant* context on demand — replacing naive full-text scanning (e.g. `grep`) with semantic retrieval. Attributed in the corpus to Andrej Karpathy's "knowledge LM wikis" idea, it treats an agent's accumulated knowledge as a queryable store rather than a flat pile of files, and is typically implemented either as a Retrieval-Augmented Generation (RAG) system or as an agent-maintained, cross-referenced set of markdown files.

## Key Mechanics

- Motivation: `grep`-style retrieval is inefficient and imprecise for feeding an agent the right context; a knowledge wiki indexes information for relevance instead of scanning for literal matches.
- Implemented as RAG: vector embeddings for semantic search, hybrid ranking (reciprocal rank fusion / RRF) combining lexical and vector signals, and chunking of documents into retrievable units.
- Concrete storage: Postgres with the pgvector extension serving as the vector store behind retrieval.
- Two-folder authored variant: a `raw` folder holds source documents the agent reads but never edits, and a `wiki` folder is agent-owned, where it writes and maintains cross-referenced markdown pages — a graph of linked notes that can be browsed and visualized (e.g. in Obsidian) as a navigable "second brain."
- **The schema as the third architectural layer**: beyond `raw` (read-only sources) and `wiki` (agent-maintained pages), an explicit *schema* layer is a standing *rules document* that dictates how the agent structures the wiki, handles a new source, formats pages, and answers questions. In the Claude Code instance this schema *is* a `claude.md` file at the vault root — making the wiki's behavior governed by a documented contract rather than improvised per run, the knowledge-base counterpart to `[[claude-md]]` as the agent's standing instruction file.
- Three-operation lifecycle (the no-embedding, agent-maintained variant): the wiki runs on **Ingest** (a new source — e.g. a YouTube link or blog post — is auto-summarized into a wiki page, existing related pages are *updated* with the new information, new concept pages are created and interlinked, and contradictions are flagged, with the index updated), **Query** (the agent answers from the processed pages, with citations, drawing across multiple interlinked pages where a single query spans several sources), and **Lint** (a periodic maintenance pass that removes contradictions and keeps knowledge current — see `[[knowledge-base-linting]]`). Some variants add an explicit `index`/schema layer alongside `raw` and `wiki`, making the structure three-tiered (raw → wiki → index/schema) rather than two-folder, and a generated `outputs` tier can hold the reports and decision files produced by querying the wiki.
- Cumulative intelligence on ingest: a second source is not merely added as new pages — the agent *intelligently updates existing pages* with the new material (e.g. enriching a neighborhood page with food information from a second guide), so the wiki's connections and synthesis grow richer with each source rather than accreting disconnected entries.
- Packaged ingestion pipeline: the authored variant can be wrapped in an installable, cross-harness skill that scaffolds the vault and exposes Ingest/Query/Lint as commands, with source capture automated by a browser web-clipper (pages, images) and transcript extractors feeding the `raw` folder, and the ingest pass put on a continuous loop for hands-off updating.
- It is a persistent, curated context source — the durable counterpart to per-run `[[context-engineering]]`, and a knowledge analogue of `[[claude-md]]` scaled from a single file to a searchable corpus.

## How It Appears in the Corpus

The Garry Tan / Y Combinator "Tokenmaxxing" talk describes G-Brain, inspired by Karpathy's "knowledge LM wikis," built to replace inefficient grep-based context retrieval. Reusing techniques from Garry's List (vector embedding, hybrid RRF, chunking), it grew into a full RAG system on Postgres + pgvector for Open Claude — cited as evidence of a "golden age of open source" where existing solutions are rapidly adapted with AI-powered tooling.

The Simon Scrapes memory-systems comparison frames the LLM Wiki pattern as its Level 5, for building an interconnected knowledge base from consumed content (articles, videos, podcasts, notes). It describes Karpathy's two-folder `raw`/`wiki` layout — read-only sources plus an agent-maintained, cross-linked markdown wiki browsable in Obsidian — and a hosted alternative, Recall, that automates the setup via a browser extension, summarizing and tagging content into a knowledge graph.

The Garry Tan / Diana Hu Stanford CS153 lecture re-presents G-Brain as an explicit three-layer memory system — a knowledge wiki, vector search, and a graph database, with a planned "epistemology" layer to capture and track nascent "hunches" that mature into groundbreaking ideas — and maps it onto a company primitive: the brain *is* the organization's internal process and accumulated knowledge inside an `[[ai-native-company]]`. This stratified design (semantic wiki + vector retrieval + relationship graph) echoes the tiered structure of `[[persistent-agent-memory]]` and the relationship layers of `[[codebase-knowledge-graph]]`.

The Nate B Jones ("AI News & Strategy Daily") comparison reframes Karpathy's two-folder wiki as an *ingest-time* synthesis system — the AI compiles understanding once as sources arrive, updating pages, flagging contradictions, and building cross-references at ingestion so later queries are cheap — and contrasts it with Open Brain's query-time approach, the distinction generalized in `[[ingest-vs-query-time-synthesis]]`. It recasts the AI's role from a one-off "Oracle" into a persistent "Maintainer" of the artifact, and flags the wiki's defining hazard: an unmaintained synthesis can become *active misinformation* whose confident prose hides outdated or inaccurate claims that have drifted from the raw source.

The Nuno Tavares "6 Claude Code GitHub Repos" overview surfaces the *integration* side of the pattern: "Obsidian Skills" gives an agent native read/write access to an existing Obsidian "second brain," speaking Obsidian's own markdown dialect (wiki links, embeds, callouts, JSON Canvas) so the agent works the knowledge base directly instead of by manual copy-paste. Where the variants above concern *building or querying* a knowledge wiki, this concerns *connecting* an agent to a human-curated one in its native format — an `[[agent-native-infrastructure]]` move applied to personal knowledge management, eliminating the copy-paste seam between a PKM store and the agent.

The 실밸개발자 ("Second Brain for AI") talk presents the LM Wiki pattern as the way to build a personal `[[second-brain]]`: a three-layer `raw`/`wiki`/`index` structure the agent manages directly *without* chunking or embedding, visualized as an Obsidian graph and automated via Claude Code. It explicitly contrasts this with RAG (chunk → embed → similarity-search → inject), arguing that because context windows have grown and agents navigate files well, the agent can now build and maintain the wiki itself rather than relying on an embedding pipeline. It frames the wiki as running three operations — Ingest, Query, and `[[knowledge-base-linting|Lint]]` — adds that the system grows richer the more it is queried, and stresses that the base must be continually validated (`[[custom-eval-systems]]`) and that injecting one's *unique context* is the real competitive edge as models converge.

The Corey Ganim "Claude + Karpathy's Second Brain is INSANE" tutorial is the hands-on, packaged realization of the same no-embedding authored wiki: an open-source, cross-harness "Claude Skill" (working across Claude Code, Codex, and Gemini CLI) scaffolds an Obsidian vault and runs the `raw` → `wiki` → `outputs` flow via `second brain ingest` / query / `lint` commands. A browser web-clipper scrapes pages and YouTube transcripts into the `raw` folder, ingestion summarizes and files them into a cross-linked `wiki` (with Obsidian's graph view showing the new connections), queries return context-aware answers from the collected data, and the ingest step can be driven continuously by Claude Code's loop feature — demonstrating the agent-maintained variant as a one-command install rather than a hand-wired pipeline.

The Teacher's Tech "Karpathy's LLM Wiki — Full Beginner Setup Guide" tutorial is the entry-level walkthrough of the same authored pattern, and is the corpus's clearest statement of the *three-layer* architecture: `raw` sources (read-only originals — PDFs, articles, notes), a `wiki` of AI-maintained index/concept/entity/summary markdown pages, and a *schema* (a rules document — `claude.md` for Claude Code — dictating how the agent structures the wiki, handles new sources, formats pages, and answers questions). It motivates the whole pattern against basic RAG's lack of memory (every query restarts from scratch, building on nothing) and demonstrates cumulative intelligence with a Japan trip-planning wiki: ingesting a Tokyo blog post creates and interlinks neighborhood pages (visible in Obsidian's graph view), then ingesting a food guide *updates those existing pages* rather than only adding new ones, so a later question spanning both sources is answered from the interconnected pages with citations. It frames the pattern as effective wherever knowledge accumulation over time is wanted (students, researchers, teachers, businesses onboarding new hires, personal encyclopedias) and stresses that the agent can `lint` for contradictions, outdated claims, orphan pages, and unpaged concepts but that human review remains essential, especially early.

## Tensions & Tradeoffs

- Retrieval quality is the new bottleneck: a knowledge wiki only helps if it surfaces the *relevant* chunks; poor embeddings or chunking silently feed the agent wrong context — the same "more context isn't better context" risk noted for `[[context-engineering]]` and `[[context-rot]]`.
- Build cost vs grep simplicity: a RAG pipeline (embeddings, hybrid ranking, vector DB) is far more infrastructure than `grep`, and the payoff assumes a corpus large enough that precise retrieval actually matters. The no-embedding, agent-maintained variant trades that pipeline cost for reliance on the agent's own file navigation and a large context window — cheaper to stand up (especially as a packaged one-command skill) but dependent on the model actually finding the right page.
- Freshness: like `[[compound-engineering]]`'s accumulated knowledge, an embedded index drifts stale relative to its source and must be re-indexed to stay accurate; the agent-maintained variant addresses this with a periodic `[[knowledge-base-linting|Lint]]` pass rather than re-indexing.
- Active misinformation, not just staleness: because the authored wiki variant *synthesizes* rather than stores, a stale page does not merely go missing — its confident prose can silently assert a synthesis that no longer matches the raw source, a sharper failure than freshness drift (see `[[ingest-vs-query-time-synthesis]]`), and the editorial decisions the AI makes while synthesizing can obscure nuance present in the original material. Grounding the wiki in a re-queryable structured store, or running a standing lint pass, is the corpus's mitigation.
- The index/schema layer keeps an agent-maintained wiki navigable as it grows, but it is one more artifact to keep consistent — the maintenance burden `[[knowledge-base-linting]]` exists to carry, and the reason an unmaintained wiki drifts. The schema also *bounds* the wiki's quality: because it governs how the agent ingests, formats, and answers, a thin or wrong schema produces a confidently mis-structured wiki, so the rules document is itself a quality lever, not boilerplate.
- Frictionless ingestion can outrun curation: a web-clipper plus a continuous ingest loop makes growing the `raw` folder nearly effortless, but mass-capturing unvetted material feeds the wiki low-signal or duplicative content the agent then synthesizes over — so automated capture shifts the burden onto the lint pass and the operator's judgment about what to clip, rather than removing it. The recurring "garbage in, garbage out" caveat: the wiki's value is bounded by the quality of the sources placed in `raw`.
- Native-format integration is a capability, not a guarantee of quality: an agent wired directly into a second brain (e.g. via Obsidian Skills) inherits whatever is *in* that store, so a disorganized or stale PKM feeds the agent disorganized or stale context — the retrieval and freshness caveats above apply regardless of how seamless the bridge is.
- Personal-scale ceiling and human review: the authored, agent-maintained wiki works best at *personal scale* (the corpus cites hundreds of articles), and the agent can still make synthesis or linking mistakes — so human review is necessary, especially in the early stages before the schema and the base have matured, the same "quality of the artifact bounds the trust" caveat applied to a tool aimed at individuals rather than teams.
- Ownership and scope: hosted variants such as Recall keep data on the vendor's servers and are geared toward content *consumption* rather than operational agent memory — distinguishing the knowledge wiki from `[[persistent-agent-memory]]` (operational recall) and raising the data-custody concern that `[[cross-tool-memory]]` addresses.
